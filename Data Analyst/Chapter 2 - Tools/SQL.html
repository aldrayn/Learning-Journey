SQL

Fundamentals
  • SQL, Database, and DBMS
    • Apa itu SQL?

      SQL (Structured Query Language) adalah bahasa yang digunakan untuk bekerja dengan database relasional. Dengan SQL Anda bisa:

      • menanyakan data,
      • mengambil data,
      • menambah data,
      • mengubah data,
      • menghapus data.

      SQL menjadi standar utama untuk mengelola data dalam sistem yang menggunakan tabel.

    • Apa itu Data?

      Data adalah kumpulan fakta, bisa berupa:

      • kata (nama, alamat),
      • angka (umur, saldo),
      • gambar,
      • atau bentuk digital lainnya.

      Data sangat penting bagi semua bisnis. Bank, perusahaan kartu kredit, hingga platform pembayaran menyimpan banyak data tentang Anda. Karena sangat penting, data harus:

      • aman,
      • tersimpan rapi,
      • dan bisa diakses dengan cepat.

    • Apa itu Database?

      Database adalah tempat untuk menyimpan data.
      Selain menyimpan, database juga menyediakan fungsi untuk:

      • menambah data,
      • menghapus data,
      • memperbarui data,
      • dan menanyakan data.

      Database ada di mana-mana dan bekerja di balik layar pada hampir semua aplikasi yang kita gunakan sehari-hari.

    • Jenis Database

      Relational Database (RDBMS)

      Data disimpan dalam bentuk tabel (baris dan kolom), seperti spreadsheet.
      Contoh RDBMS:

      • MySQL
      • PostgreSQL
      • Oracle Database
      • SQL Server
      • IBM DB2

      Keunggulan:

      • kuat, stabil, cocok untuk transaksi,
      • mudah di-query memakai SQL,
      • bisa membuat relasi antar tabel (foreign key).

      NoSQL Database (tidak wajib SQL, lebih fleksibel)

      Digunakan untuk data besar, tak terstruktur, real-time, atau skala sangat besar.

      Jenis-jenis NoSQL:

      • Document Store → MongoDB, CouchDB
      • Key-Value Store → Redis, DynamoDB
      • Column Store → Cassandra, HBase
      • Graph Database → Neo4j, Amazon Neptune

      Ciri:

      • fleksibel,
      • tidak butuh skema tabel,
      • scalable horizontal.

      In-Memory Database

      • Data disimpan di RAM untuk kecepatan super tinggi.
      • Contoh: Redis, Memcached.

      Data Warehouse

      • Tempat menyimpan data dalam jumlah sangat besar untuk analitik.
      • Contoh: BigQuery, Snowflake, Redshift.

      Time-Series Database

      • Menyimpan data yang berdasarkan waktu (IoT, sensor, log).
      • Contoh: InfluxDB, TimescaleDB.
    • Apa itu RDBMS?

      RDBMS (Relational Database Management System) adalah perangkat lunak yang mengelola database relasional.

      Fungsinya:

      • mengatur penyimpanan,
      • mengatur keamanan,
      • mengatur akses,
      • menjalankan SQL,
      • menjaga integritas data.

      Contoh RDBMS:

      • MySQL
      • Oracle Database
      • PostgreSQL
      • SQL Server
      • IBM DB2

      RDBMS menjadi tulang punggung aplikasi di perbankan, transportasi, kesehatan, dan industri besar lainnya.

    • Perintah SQL Dasar

      Lima perintah SQL yang paling sering digunakan:

      • CREATE → membuat tabel
      • INSERT → mengisi data
      • SELECT → menanyakan data
      • UPDATE → memperbarui data
      • DELETE → menghapus data

      Dengan lima perintah ini, sebagian besar kebutuhan dasar pengelolaan database sudah bisa dilakukan.

  • Relational Database
    • Model Relasional
      • Model relasional adalah model pengelolaan data yang menyimpan informasi dalam bentuk relation (tabel dua dimensi) yang terdiri dari tuple (baris) dan attribute (kolom).
      • Model ini mendasari sebagian besar sistem manajemen basis data modern seperti MySQL, PostgreSQL, SQL Server, dan Oracle.
      • Karakteristik Utama
        • Representasi Data dalam Tabel

          Semua data direpresentasikan dalam struktur tabel yang terdefinisi dengan jelas, sehingga mudah dipahami dan dimanipulasi.

        • Independensi Data

          Perubahan pada penyimpanan fisik tidak memengaruhi struktur logis, sehingga aplikasi tetap dapat berjalan tanpa modifikasi besar.

        • Kemudahan Perancangan dan Skalabilitas

          Struktur tabular mendukung normalisasi, ekspansi skema, dan pemrosesan kueri secara efisien.

      • Contoh

        Sebuah sistem perpustakaan dapat direpresentasikan sebagai tabel BOOK, AUTHOR, COPY, LOAN, dan BORROWER.

    • Entity–Relationship (ER) Model
      • Konsep

        Entity–Relationship Model adalah model konseptual yang digunakan dalam tahap perancangan basis data untuk memvisualisasikan entitas, atribut, dan relasi antar entitas sebelum dibangun ke dalam model relasional.

        Contoh

        Dalam sistem perpustakaan:

        • Entitas:
          • BOOK,
          • AUTHOR,
          • COPY,
          • BORROWER,
          • LOAN
        • Relasi:
          • Satu buku dapat ditulis oleh lebih dari satu penulis.
          • Satu salinan buku hanya dapat dipinjam oleh satu peminjam pada satu waktu.

        ERD menggambarkan konsep tersebut dalam bentuk diagram konseptual.

      • Entitas dan Atribut

        Entitas

        Entitas adalah objek, konsep, atau hal yang keberadaannya dapat diidentifikasi secara unik dan disimpan sebagai data dalam basis data.

        Contoh Entitas

        • BOOK
        • AUTHOR
        • BORROWER

        Dalam ERD digambarkan sebagai persegi panjang.

        Atribut

        Atribut adalah karakteristik atau properti yang dimiliki oleh suatu entitas dan menyimpan nilai spesifik tentang entitas tersebut.

        Contoh Atribut BOOK

        • TITLE
        • EDITION
        • ISBN
        • YEAR

        Dalam ERD digambarkan sebagai oval.

      • Pemetaan Entitas ke Model Relasional
        • Entitas pada ER Model dipetakan menjadi tabel.
        • Atribut pada entitas dipetakan menjadi kolom dalam tabel tersebut.
        • Setiap tuple (baris) mewakili satu instansi entitas.

        Contoh Pemetaan

        • Entitas: BOOK
        • Atribut: TITLE, EDITION, YEAR, ISBN → menjadi kolom tabel BOOK

          | TITLE | EDITION | YEAR | ISBN |

      • Tipe Data dalam Model Relasional

        Karakter

        • CHAR(n): panjang tetap
        • VARCHAR(n): panjang variabel

        Numerik

        • INT
        • DECIMAL(p, s)

        Tanggal dan Waktu

        • DATE
        • TIME
        • TIMESTAMP

        Lainnya

        • BOOLEAN
        • BLOB (binary large object seperti gambar, file)
        • JSON (khusus DB modern seperti PostgreSQL)

      • Primary Key dan Foerign Key

        Primary Key

        Primary Key (PK) adalah atribut atau kombinasi atribut yang mengidentifikasi setiap tuple dalam suatu tabel secara unik dan tidak boleh bernilai duplikat ataupun NULL.

        Fungsi

        • Menjamin integritas baris (tidak ada baris yang identik).
        • Menjadi referensi bagi tabel lain dalam relasi.

        Contoh

        • AUTHOR_ID
        • BOOK_ID

        Foreign Key

        Foreign Key (FK) adalah atribut dalam suatu tabel yang merujuk pada Primary Key dari tabel lain untuk membentuk hubungan antar tabel.

        Contoh

        • COPY.BOOK_ID → mereferensikan BOOK.BOOK_ID
        • LOAN.COPY_ID → mereferensikan COPY.COPY_ID

        FK memastikan setiap baris memiliki keterkaitan logis dengan tabel lain dan menjaga referential integrity.

      • Relasi dalam Database

        One-to-One (1 : 1)

        • Satu baris di tabel A hanya boleh punya satu pasangan di tabel B, dan sebaliknya.
        • Contoh:
          • Satu orang hanya punya satu KTP.
          • Satu user hanya punya satu profil detail.
        • Karakteristik:
          • Biasanya dilakukan untuk memecah tabel besar agar lebih rapi (normalisasi).
          • Menggunakan foreign key unik.
          • Salah satu tabel menyimpan foreign key yang juga diberi UNIQUE constraint.
        • Contoh Struktur
          User (id PK)
UserDetail (id PK, user_id FK UNIQUE)

        One-to-Many (1 : N)

        • Satu entitas A dapat berhubungan dengan banyak entitas B.
          Tapi entitas B hanya boleh punya satu referensi balik ke A.
        • Contoh:
          • Satu kategori punya banyak produk.
          • Satu dosen membimbing banyak mahasiswa.
        • Karakteristik:
          • Tabel “banyak” (child) menyimpan foreign key ke tabel “satu” (parent).
          • Foreign key tidak perlu unik, karena banyak baris bisa memakai FK yang sama.
        • Contoh struktur:
          Category (id PK)
Product (id PK, category_id FK)

        Many-to-Many (M : N)

        • Satu entitas A bisa punya banyak entitas B, dan satu entitas B bisa punya banyak entitas A.
        • Contoh:
          • Mahasiswa mengambil banyak mata kuliah.
          • Produk muncul di banyak order, dan satu order punya banyak produk.
        • Karakteristik:
          • Tidak bisa langsung dibuat hanya dengan foreign key.
          • BUTUH tabel penghubung (junction table / associative table / linking table).
          • Tabel relasi berisi dua foreign key yang masing-masing menunjuk ke tabel A dan B.
          • Sering menggunakan composite primary key (gabungan dua FK).
        • Contoh Struktur
          Student (id PK)
Course (id PK)
StudentCourse (student_id FK, course_id FK, PRIMARY KEY(student_id, course_id))

  • DDL vs DML

    Dalam database relasional, SQL digunakan untuk berinteraksi dengan entitas (tabel), atribut (kolom), dan tupel (baris).
    Semua pernyataan SQL terbagi menjadi dua kategori utama:

    • Data Definition Language (DDL)

      DDL adalah pernyataan SQL yang digunakan untuk mendefinisikan, mengubah, atau menghapus struktur database, seperti tabel dan objek lain.

      Pernyataan DDL yang Umum:

      • CREATE
        • Digunakan untuk membuat objek database baru, seperti tabel, beserta struktur kolomnya.
        • Contoh: membuat tabel AUTHOR.
      • ALTER
        • Digunakan untuk memodifikasi struktur objek yang sudah ada.
        • Contoh: menambah kolom, menghapus kolom, atau mengubah tipe data.
      • TRUNCATE
        • Menghapus seluruh data dalam tabel tanpa menghapus tabelnya.
        • Lebih cepat daripada DELETE.
      • DROP

        Menghapus seluruh objek database, termasuk tabel dan struktur kolomnya.

      Inti DDL:

      Berhubungan dengan struktur database, bukan datanya.

    • Data Manipulation Language (DML)

      DML digunakan untuk membaca dan memodifikasi data di dalam tabel.

      Secara umum dikenal sebagai operasi CRUD:

      • Create
      • Read
      • Update
      • Delete

      Pernyataan DML yang Umum:

      • INSERT
        • Menambahkan baris data baru ke dalam tabel.
        • Bisa satu baris atau banyak baris sekaligus.
      • SELECT
        • Membaca atau memilih data dari tabel.
        • Bisa mengambil sebagian kolom, baris tertentu, atau seluruh tabel.
      • UPDATE

        Mengubah nilai data pada satu atau beberapa baris.

      • DELETE

        Menghapus satu atau beberapa baris dari tabel.

      Inti DML:

      Berhubungan dengan data, bukan struktur tabelnya.

  • Statement
    • DDL
      • CREATE TABLE

        CREATE TABLE adalah pernyataan Data Definition Language (DDL) yang digunakan untuk membuat tabel baru dalam database relasional.
        Pernyataan ini menetapkan:

        • Nama tabel
        • Nama kolom
        • Tipe data setiap kolom
        • Opsi atau constraints tambahan (misalnya PRIMARY KEY, NOT NULL, dsb.)

        Struktur Umum Sintaks

        CREATE TABLE nama_tabel (
    nama_kolom1 tipe_data [opsi],
    nama_kolom2 tipe_data [opsi],
    ...
);

        Keterangan:

        • Setiap definisi kolom dipisahkan koma.
        • Opsi tambahan bisa berupa:
          • PRIMARY KEY
          • NOT NULL
          • UNIQUE
          • CHECK
          • DEFAULT, dll.

        Contoh 1 — Tabel Provinsi Kanada

        CREATE TABLE province (
    id CHAR(2) PRIMARY KEY NOT NULL,
    name VARCHAR(24)
);

        Penjelasan:

        • id bertipe CHAR(2) → panjang tetap, contoh: ‘AB’, ‘BC’.
        • name bertipe VARCHAR(24) → panjang variabel, contoh: “ALBERTA”.
        • PRIMARY KEY memastikan id unik dan tidak boleh duplikat.
        • NOT NULL memastikan kolom tidak boleh berisi nilai kosong.

        Tabel ini menghasilkan 2 kolom:

        • id untuk kode provinsi
        • name untuk nama lengkap provinsi

        Contoh 2 — Tabel AUTHOR (Database Perpustakaan)

        CREATE TABLE author (
    author_id CHAR(2) PRIMARY KEY NOT NULL,
    lastname VARCHAR(15) NOT NULL,
    firstname VARCHAR(15) NOT NULL,
    email VARCHAR(40),
    city VARCHAR(15),
    country CHAR(2)
);

        Penjelasan penting:

        • PRIMARY KEY: author_id
          → memastikan setiap baris mewakili satu penulis unik.
        • NOT NULL: lastname, firstname
          → setiap penulis wajib memiliki nama depan dan nama belakang.
        • Kolom lain bisa bernilai NULL (opsional).

        Catatan:

        • CREATE TABLE membuat tabel baru berdasarkan entitas dalam ERD.
        • Nama entitas → nama tabel.
        • Nama atribut → nama kolom.
        • Setiap kolom wajib ditentukan tipe datanya.
        • Constraint seperti PRIMARY KEY atau NOT NULL membantu menjaga kualitas dan konsistensi data.
        • Struktur tabel dituliskan dalam tanda kurung dan dipisahkan koma per kolom.

        Tambahan - Tabel dengan Foreign Key

        CREATE TABLE nama_tabel (
    kolom1 tipe_data,
    kolom2 tipe_data,
    kolom_fk tipe_data,
    FOREIGN KEY (kolom_fk) REFERENCES tabel_lain(kolom_pk)
);

        Contoh:

        • Tabel mahasiswa
          CREATE TABLE student (
    student_id INT PRIMARY KEY,
    name VARCHAR(50)
);
        • Tabel mata kuliah
          CREATE TABLE course (
    course_id INT PRIMARY KEY,
    title VARCHAR(100)
);
        • Tabel penghubung (student_course)
          CREATE TABLE student_course (
    student_id INT,
    course_id INT,
    PRIMARY KEY (student_id, course_id),
    FOREIGN KEY (student_id) REFERENCES student(student_id),
    FOREIGN KEY (course_id)   REFERENCES course(course_id)
);

      • ALTER

        ALTER TABLE digunakan untuk mengubah struktur tabel yang sudah ada.

        Fungsi utama:

        • Menambah kolom
          ALTER TABLE authors ADD COLUMN phone_number BIGINT;
        • Mengubah nama kolom
          ALTER TABLE authors RENAME COLUMN firstname TO first_name;
        • Mengubah tipe data kolom
          ALTER TABLE authors MODIFY phone_number CHAR(20);
        • Menghapus kolom
          ALTER TABLE authors DROP COLUMN phone_number;
        • Menambah/menghapus constraint (PK, FK, UNIQUE, dll.)
          • Tambah PRIMARY KEY
            ALTER TABLE authors ADD CONSTRAINT pk_auth PRIMARY KEY (author_id);
          • Tambah FOREIGN KEY
            ALTER TABLE books ADD CONSTRAINT fk_book_author
FOREIGN KEY (author_id) REFERENCES authors(author_id);
          • Tambah UNIQUE
            ALTER TABLE authors ADD CONSTRAINT unique_email UNIQUE (email);
          • Hapus constraint
            ALTER TABLE authors DROP INDEX unique_email;

            Keterangan:

            • pk_auth, fk_book_author, dll itu nama aturan (nama constraint), bukan kolom.
            • Supaya jika nanti kamu ingin menghapus, mengubah, atau mendiagnosis aturan ini, kamu bisa merujuknya dengan nama tersebut.
        • Mengubah constraint NOT NULL
          • Menambah NOT NULL:
            ALTER TABLE authors MODIFY email VARCHAR(40) NOT NULL;
          • Menghapus NOT NULL:
            ALTER TABLE authors MODIFY email VARCHAR(40) NULL;
        • Mengatur atau menghapus DEFAULT value
          • Menambah DEFAULT:
            ALTER TABLE MODIFY country CHAR(2) DEFAULT 'ID';
          • Menghapus DEFAULT:
            ALTER TABLE MODIFY country DROP DEFAULT;
        • Mengubah nama tabel
          ALTER TABLE authors RENAME TO writers;

        Catatan:

        • Mengubah tipe data kolom yang sudah berisi data bisa menimbulkan error jika datanya tidak kompatibel.
        • Setiap baris dalam perintah ALTER biasanya melakukan satu perubahan.

      • DROP

        DROP TABLE digunakan untuk menghapus seluruh tabel dari database, termasuk seluruh datanya.

        Contoh

        DROP TABLE authors;

        Catatan:

        • Setelah dieksekusi, tabel hilang permanen.
        • Biasanya tidak bisa di-rollback jika DB tidak mendukung transaksi DDL.

      • TRUNCATE

        TRUNCATE TABLE digunakan untuk menghapus seluruh baris dalam tabel, tetapi tidak menghapus tabelnya.

        Contoh:

        TRUNCATE TABLE authors IMMEDIATE;

        Catatan:

        • Jauh lebih cepat daripada DELETE FROM table tanpa WHERE.
        • Menghapus semua baris tanpa memindai satu per satu.
        • Tidak bisa dibatalkan jika menggunakan IMMEDIATE.

    • DML
      • SELECT
        • Konsep
          • Tujuan utama RDBMS bukan hanya menyimpan data, tetapi juga mengambil data dengan cepat dan akurat.
          • Untuk itu digunakan pernyataan SELECT, yang termasuk dalam kategori DML (Data Manipulation Language).
          • DML digunakan untuk:
            • membaca data → SELECT
            • menambah data → INSERT
            • memperbarui data → UPDATE
            • menghapus data → DELETE
          • Pernyataan SELECT disebut query, dan outputnya disebut result set atau tabel hasil.

        • SELECT Dasar
          • Bentuk paling sederhana:
            SELECT * FROM nama_tabel;

            Keterangan:

            \* berarti mengambil semua kolom.

            Contoh tabel book berisi empat baris data.

            SELECT * FROM book menampilkan semua baris dan semua kolom.

          • Anda juga bisa mengambil hanya kolom tertentu:
            SELECT book_id, title FROM book;

            Hal penting:

            Urutan kolom mengikuti urutan di pernyataan SELECT, bukan urutan di tabel.

        • Membatasi Hasil dengan WHERE
          • Jika ingin mengambil data berdasarkan kondisi, gunakan WHERE.
          • Contoh kasus: mencari judul buku dengan book_id = 'B1'.
            SELECT title, book_id
FROM book
WHERE book_id = 'B1';
          • Klausa WHERE menggunakan predikat.
          • Predikat adalah kondisi yang menghasilkan nilai TRUE, FALSE, atau UNKNOWN.
          • Hanya baris yang kondisi WHERE-nya TRUE yang akan muncul di result set.

        • Operator Perbandingan

          RDBMS mendukung beberapa operator untuk membangun predikat:

          • = → sama dengan
          • > → lebih besar dari
          • < → kurang dari
          • >= → lebih besar atau sama dengan
          • <= → kurang dari atau sama dengan
          • <> atau != → tidak sama dengan

          Operator ini digunakan dalam klausa WHERE untuk memfilter data.

        • Tambahan
          • Nilai UNKNOWN dalam predikat
            • Jika kolom berisi NULL, predikat bisa menghasilkan UNKNOWN, bukan TRUE/FALSE.
            • Baris dengan hasil UNKNOWN tidak muncul kecuali menggunakan IS NULL.
          • SELECT tanpa FROM

            Beberapa RDBMS mengizinkan select tanpa from untuk ekspresi sederhana.

            • Konstanta
              SELECT 'Hello World';

              Output:

              Hello World

            • Perhitungan
              SELECT 10 * 5;

              Output:

              50

            • Fungsi
              SELECT CURRENT_DATE;

              Output:

              2025-11-16

          • SELECT DISTINCT

            Untuk menghilangkan duplikasi:

            SELECT DISTINCT title FROM book;
          • Alias kolom
            SELECT title AS book_title FROM book  ;
          • Urutan eksekusi SQL

            Agar memahami cara kerja SELECT:

            • FROM
            • WHERE
            • SELECT

      • INSERT
        • Konsep
          • Pernyataan INSERT digunakan untuk memasukkan atau menambahkan baris baru ke dalam tabel database relasional.
          • INSERT termasuk ke dalam DML (Data Manipulation Language), yaitu kelompok perintah untuk membaca dan memodifikasi data (SELECT, INSERT, UPDATE, DELETE).

          Kapan INSERT digunakan?

          • Setelah tabel dibuat menggunakan pernyataan DDL seperti CREATE TABLE, tabel tersebut harus diisi data.
          • Untuk itu digunakan pernyataan INSERT.

          Sintaks Dasar INSERT

          Struktur dasar pernyataan:

          INSERT INTO nama_tabel (kolom1, kolom2, kolom3, ...)
VALUES (nilai1, nilai2, nilai3, ...);

          Penjelasan:

          • nama_tabel → tabel yang akan diisi
          • daftar kolom → kolom yang akan diberi nilai
          • VALUES → bagian yang berisi data yang akan dimasukkan
          • Jumlah kolom harus sama dengan jumlah nilai di VALUES

        • Contoh

          Kasus — Tabel Penulis

          Misalkan tabel authors (penulis) memiliki 6 kolom:

          • author_id
          • lastname
          • firstname
          • email
          • city
          • country

          Untuk menambahkan baris data penulis Raul Chong:

          INSERT INTO authors (author_id, lastname, firstname, email, city, country)
VALUES ('A', 'Chong', 'Raul', 'RFC@IBM.com', 'Toronto', 'CA');

          Setiap kolom diberi nilai sesuai urutan kolom.

          Menambahkan Banyak Baris Sekaligus

          Anda tidak harus meng‐insert satu baris saja. SQL mendukung penyisipan beberapa baris dalam satu statement:

          INSERT INTO authors (author_id, lastname, firstname, email, city, country)
VALUES
('A', 'Chong', 'Raul', 'RFC@IBM.com', 'Toronto', 'CA'),
('B', 'Ahuja', 'Rav', 'rav@ibm.com', 'Toronto', 'CA');

          Setiap baris dipisahkan dengan tanda koma.

        • Tambahan

          INSERT tanpa daftar kolom

          Jika ingin memasukkan nilai untuk semua kolom dan urutannya cocok dengan struktur tabel:

          INSERT INTO authors
VALUES ('A', 'Chong', 'Raul', 'RFC@IBM.com', 'Toronto', 'CA');

          Namun ini tidak disarankan karena rawan error jika struktur tabel berubah.

          Null dan kolom opsional

          Jika kolom tertentu boleh NULL, Anda bisa menuliskan:

          INSERT INTO authors (author_id, lastname)
VALUES ('A', 'Chong');

          Kolom lain akan menjadi NULL secara default.

          INSERT SELECT (memindahkan data antar tabel)

          Cara ketiga untuk memasukkan data (tidak dibahas di video tapi penting):

          INSERT INTO new_table
SELECT * FROM old_table;

          Berguna untuk duplikasi data atau import antar tabel.

      • UPDATE
        • Konsep

          Fungsi - Mengubah Isi Data

          Untuk mengubah nilai tertentu pada baris tertentu di dalam tabel.

          Sintaks Umum

          UPDATE nama_tabel
SET nama_kolom = nilai_baru,
    nama_kolom2 = nilai_baru2
WHERE kondisi;

          Catatan

          • HARUS pakai WHERE kalau tidak mau seluruh tabel berubah.
          • Tanpa WHERE → semua baris akan kena update.

        • Contoh

          Ada tabel AUTHOR dan kita ingin mengubah data penulis dengan AUTHOR_ID = 'A2'.

          Misal sebelumnya:

          AUTHOR_IDFIRSTNAMELASTNAME
          A1AlKhufra
          A2RavAhuja

          Perintah UPDATE:

          UPDATE AUTHOR

  • Complement
    • String Pattern & Range
      • Pattern Matching

        Definisi

        Pola string digunakan ketika nilai yang dicari tidak diketahui secara persis, tetapi diketahui sebagian polanya. SQL menyediakan mekanisme pencocokan pola melalui operator LIKE dan beberapa wildcard (karakter pengganti).

        Konsep

        • Operator LIKE

          Digunakan dalam klausa WHERE untuk mencocokkan nilai string terhadap pola tertentu.

          Sintaks:

          kolom LIKE 'pola'

          LIKE mendukung dua wildcard utama:

          • % → nol atau lebih karakter
          • _ → tepat satu karakter

        • Wildcard % (Any-Length Wildcard)

          Karakter % digunakan untuk mencocokkan 0 atau lebih karakter.

          • Pola umum dan contoh:
            PolaArtiContoh
            'R%'dimulai dengan huruf R'Rina', 'Ravi'
            '%son'diakhiri “son”'Jackson', 'Anderson'
            '%art%'mengandung kata "art"'Artbook', 'Smartphone'
            'D%g'dimulai D, diakhiri g'Dong', 'Drag'
          • Contoh:
            SELECT * FROM authors
WHERE first_name LIKE 'R%';

        • Wildcard _ (Single-Character Wildcard)

          Karakter _ mewakili tepat satu karakter.

          • Pola umum dan contoh:
            PolaArtiContoh
            'M_n'tiga huruf, mulai M, akhir n'Man', 'Min', 'Mun'
            'A__a'empat huruf, mulai A, akhir a'Alya', 'Aria'
            'J___'empat huruf, mulai J'John', 'Jane'
            '__y'tiga huruf, akhir y'Ray', 'Jay'
          • Contoh:
            SELECT * FROM authors
WHERE first_name LIKE 'M_n';

        • Kombinasi Wildcard

          Pola string dapat menggabungkan % dan _ untuk pencarian lebih spesifik.

          Contoh:

          • Dua huruf pertama bebas, lalu ada “an”, lalu bebas:

            LIKE '__an%' → cocok 'Joanne', 'Brandon'

          • Dimulai huruf “S”, diikuti tepat dua karakter, lalu bebas:

            LIKE 'S__%' → 'Sara', 'Sammy'

          • Dimulai “A”, huruf ke-3 adalah “i”:

            LIKE 'A_i%' → 'Aida', 'Arif'

        • Case Sensitivity

          Perilaku LIKE terhadap huruf besar/kecil bergantung pada database:

          • MySQL: biasanya case-insensitive pada collation default.
          • PostgreSQL: case-sensitive.
          • SQL Server: tergantung collation.

          Jika ingin pasti case-insensitive:

          WHERE LOWER(name) LIKE LOWER('r%');

        • Escape Character (Menghindari Wildcard)

          Jika ingin mencari karakter literal % atau _, gunakan ESCAPE.

          Contoh:

          SELECT comment
FROM reviews
WHERE comment LIKE '%\\%%' ESCAPE '\\';

          Contoh di atas mencari komentar yang benar-benar mengandung karakter %.

        • Batasan LIKE

          LIKE tidak efisien untuk pencarian kompleks seperti:

          • pencarian fuzzy
          • typo tolerance
          • stemming (book → books)

          Untuk itu digunakan:

          • REGEXP / RLIKE (regex)
          • Full-text search

        Contoh

        # Cari nama belakang yang huruf ketiga adalah “r”
WHERE last_name LIKE '__r%'

# Cari buku yang judulnya terdiri dari 5 huruf
WHERE title LIKE '_____'

# Cari email dengan domain Gmail
WHERE email LIKE '%@gmail.com'

# Cari penulis yang nama depannya mengandung “av” (dimanapun)
WHERE first_name LIKE '%av%'

      • Range Values with BETWEEN

        Konsep

        Operator BETWEEN digunakan dalam klausa WHERE untuk memfilter data berdasarkan rentang nilai minimum hingga maksimum. Rentang ini bersifat inklusif, artinya nilai batas bawah dan batas atas ikut disertakan dalam hasil.

        Fungsi Utama BETWEEN

        Digunakan ketika nilai yang ingin dicari berada di dalam satu rentang kontinu, seperti:

        • rentang angka,
        • rentang tanggal,
        • rentang waktu,
        • rentang teks (secara alfabetis).

        Sintaks dasar:

        kolom BETWEEN nilai1 AND nilai2

        BETWEEN bersifat inklusif

        Artinya:

        BETWEEN 10 AND 20
≡ (nilai >= 10 AND nilai <= 20)

        Contoh:

        SELECT * FROM books
WHERE pages BETWEEN 290 AND 300;

        Maka halaman 290 dan 300 disertakan.

        Contoh

        • BETWEEN untuk Angka

          Cocok untuk data numerik seperti harga, usia, skor, jumlah halaman, jumlah stok, dsb.

          # Pelanggan dengan total belanja 500 ribu sampai 1 juta
WHERE total_purchase BETWEEN 500000 AND 1000000;
        • BETWEEN untuk Tanggal

          BETWEEN sangat sering digunakan untuk rentang waktu, misalnya satu minggu, satu bulan, satu tahun.

          # Ambil data kuartal pertama 2025
WHERE date BETWEEN '2025-01-01' AND '2025-03-31';
        • BETWEEN untuk Waktu (Time)
          # Cari aktivitas yang terjadi antara jam 06–12
WHERE event_time BETWEEN '06:00' AND '12:00';
        • BETWEEN untuk Teks (Alphabetical Range)

          Digunakan untuk rentang alfabet.

          # Ambil nama yang alfabetnya mulai dari huruf A sampai M
WHERE last_name BETWEEN 'A' AND 'M';

# Ambil nama produk antara 'Dell' dan 'Samsung'
WHERE product_name BETWEEN 'Dell' AND 'Samsung';
        • Penulisan Rentang yang Menggunakan Urutan Terbalik

          Beberapa DBMS tidak mengizinkan BETWEEN dengan nilai terbalik (nilai2 < nilai1).

          Beberapa lainnya masih menerima tapi hasilnya kosong.

          Hindari contoh seperti:

          WHERE age BETWEEN 50 AND 20;  -- salah
        • BETWEEN dengan NOT (NOT BETWEEN)

          Digunakan untuk mengambil nilai di luar rentang.

          Contoh:

          # Ambil nilai di bawah 50 atau di atas 90:
WHERE score NOT BETWEEN 50 AND 90;

# Ambil produk yang harganya bukan 100–500:
WHERE price NOT BETWEEN 100 AND 500;

        Batasan Penggunaan BETWEEN

        BETWEEN kurang cocok ketika:

        • nilai yang dicari tidak berurutan,

          contoh: mencari negara tertentu → lebih cocok menggunakan IN

        • Anda membutuhkan rentang tidak inklusif (eksklusif):

          contoh: lebih dari 10 dan kurang dari 20 → gunakan > dan <

        Contoh eksklusif (tanpa BETWEEN):

        WHERE age > 10 AND age < 20;

      • Sets of Values (IN)

        Konsep

        Operator IN digunakan untuk memfilter data berdasarkan sekumpulan nilai tertentu. IN sangat berguna ketika nilai yang dicari tidak bisa dikelompokkan dalam rentang, atau ketika penggunaan banyak kondisi OR membuat query menjadi panjang dan tidak efisien.

        Fungsi Utama IN

        Digunakan untuk membandingkan sebuah kolom dengan daftar nilai yang eksplisit.

        Sintaks dasar:

        kolom IN (nilai1, nilai2, nilai3, ...)

        Ini setara dengan:

        kolom = nilai1 OR kolom = nilai2 OR kolom = nilai3 ...

        Kapan Menggunakan IN

        Gunakan IN bila:

        • nilai yang dicari adalah beberapa nilai spesifik, bukan rentang,
        • kolom berisi kategori atau label tetap, seperti negara, jurusan, status order,
        • ingin menghindari penulisan OR yang panjang.

        Contoh kasus:

        • Negara: 'Canada', 'India', 'China'
        • Jurusan: 'CS', 'Math', 'Economics'
        • Status order: 'Pending', 'Shipped', 'Returned'

        Contoh

        • IN dengan Nilai Teks

          Digunakan untuk mencocokkan beberapa kategori teks.

          # Pilih departemen tertentu
WHERE department IN ('Finance', 'HR', 'IT');
        • IN dengan Nilai Angka

          Selain teks, IN juga dapat digunakan untuk angka.

          WHERE rating IN (3, 4, 5);
        • IN dengan Nilai Tanggal

          Tanggal tertentu dalam satu daftar juga bisa disaring dengan IN.

          # Hari libur nasional
WHERE holiday_date IN ('2025-05-01', '2025-08-17', '2025-12-25');
        • IN Bersama Subquery

          IN sering digunakan dengan subquery untuk perbandingan dinamis.

          # Ambil buku yang ditulis oleh penulis yang berasal dari India.

WHERE author_id IN (
    SELECT author_id
    FROM authors
    WHERE country = 'India'
);

# Ambil siswa yang mengikuti mata kuliah tertentu

WHERE student_id IN (
    SELECT student_id
    FROM enrollments
    WHERE course_id = 'CS101'
);

# Ambil transaksi yang dilakukan oleh pelanggan VIP

WHERE customer_id IN (
    SELECT id FROM customers WHERE tier = 'VIP'
);
        • NOT IN

          Digunakan untuk mengecualikan daftar nilai.

          Contoh:

          # Barang yang bukan kategori 1, 2, atau 3:
WHERE category_id NOT IN (1, 2, 3);

        NULL

        • Konsep

          NULL Bukan Angka, Bukan String, dan Bukan Nol

          • NULL = nilai tidak diketahui, bukan kosong, bukan zero.
          • Karena itu, NULL tidak pernah dianggap sama atau tidak sama dengan nilai apa pun, termasuk NULL itu sendiri.

          Perbandingan Apa Pun dengan NULL → UNKNOWN

          Jika kamu melakukan perbandingan seperti:

          • 5 = NULL → UNKNOWN
          • NULL = NULL → UNKNOWN
          • NULL IN (1, 2, 3) → UNKNOWN
          • 1 IN (NULL) → UNKNOWN

          Database tidak tahu apakah NULL itu sebenarnya mungkin berarti 1, 2, atau yang lain → tidak bisa dipastikan → hasilnya UNKNOWN.

          Mengapa UNKNOWN Jadi Masalah?

          Dalam klausa WHERE:

          • Hanya baris yang kondisi WHERE-nya TRUE yang dikembalikan.
          • Kondisi yang FALSE dan UNKNOWN → dibuang.

          Artinya:

          WHERE id IN (1, 2, NULL);
          • jika id bernilai 1 → TRUE → masuk hasil
          • jika id bernilai 2 → TRUE → masuk hasil
          • jika id bernilai 5 → UNKNOWN (karena dibandingkan dengan NULL) → tidak masuk

          Masalah muncul ketika subquery menghasilkan NULL, karena itu membuat kondisi IN menghasilkan banyak UNKNOWN.

        • Contoh

          Contoh Kasus Bermasalah - Subquery berisi NULL

          SELECT * FROM orders
WHERE customer_id IN (
    SELECT customer_id FROM customers
);  -- kolom customer_id mengandung NULL

          Jika tabel customers punya baris:

          customer_id
-----------
1
2
NULL

          Maka hasil WHERE menjadi ambiguous, dan bisa menyebabkan:

          • baris yang harusnya muncul → hilang
          • hasil query menjadi tidak akurat

          Contoh Dampak Unpredictable Result

          Misal data:

          orders.customer_id = 5

          Cek:

          5 IN (1, 2, NULL) → UNKNOWN

          Mari pecah satu per satu:

          PerbandinganHasil
          5 = 1FALSE
          5 = 2FALSE
          5 = NULLUNKNOWN

          Lalu SQL menggabungkan hasil itu, kira-kira jadi seperti:

          FALSE OR FALSE OR UNKNOWN

          Dalam logika 3-nilai SQL:

          • TRUE OR apa pun = TRUE
          • FALSE OR FALSE = FALSE
          • FALSE OR UNKNOWN = UNKNOWN

          Maka hasilnya adalah:

          FALSE OR FALSE OR UNKNOWN = UNKNOWN

          lalu kita tahu bahwa dalam WHERE:

          • TRUE → baris diambil
          • FALSE → baris dibuang
          • UNKNOWN → baris dibuang

          Karena hasil 5 IN (1,2,NULL) = UNKNOWN

          • baris dengan customer_id = 5 dibuang
          • tidak muncul di hasil query

          Solusi

          • Buang NULL dari subquery
            WHERE customer_id IN (
    SELECT customer_id
    FROM customers
    WHERE customer_id IS NOT NULL
);
          • Gunakan EXISTS (recommended best practice)

            Karena EXISTS tidak peduli nilai NULL.

            SELECT emp.id, emp.name
FROM employees emp
WHERE EXISTS (
    SELECT 1
    FROM employees e
    WHERE e.manager_id = emp.id
);

            EXISTS hanya memeriksa keberadaan baris, bukan nilai literalnya.

          • Jika nilai NULL memang sengaja dibiarkan

            Gunakan COALESCE (mengubah NULL → nilai lain):

            WHERE COALESCE(customer_id, -1) IN (1, 2, 3);

            Tapi ini hanya dipakai kalau kamu paham benar konsekuensinya.

    • Sorting (ORDER BY)
      • Konsep
        • ORDER BY digunakan untuk mengurutkan kumpulan hasil (result set) dari pernyataan SELECT.
        • Tujuannya agar data tampil lebih teratur dan mudah dibaca.
        • Tanpa ORDER BY, data biasanya muncul tanpa urutan khusus (tergantung penyimpanan fisik).
        • Contoh dasar:
          SELECT *
FROM book
ORDER BY title;

        Urutan Menaik (Ascending – ASC)

        • Secara default, SQL akan mengurutkan data dalam urutan menaik (ASC).
        • Urutan menaik berarti:
          • Angka kecil → angka besar
          • Huruf A → Z
          • Tanggal lama → tanggal baru
        • Contoh:
          SELECT title
FROM book
ORDER BY title ASC;

          Karena ASC adalah default, penulisannya boleh disederhanakan:

          ORDER BY title;

        Urutan Menurun (Descending – DESC)

        • Untuk mengurutkan dari besar → kecil, atau Z → A, gunakan DESC.
        • Contoh:
          SELECT title
FROM book
ORDER BY title DESC;

          Contoh untuk angka:

          SELECT title, pages
FROM book
ORDER BY pages DESC;

      • Tambahan

        Urut Berdasarkan Lebih dari Satu Kolom

        • Gunakan beberapa kolom di ORDER BY.
        • Kolom pertama menjadi prioritas utama, kemudian kolom berikutnya.
        • Contoh:
          ORDER BY last_name ASC, first_name ASC;

          Contoh kompleks:

          ORDER BY country ASC, pages DESC;

        Menggunakan Nomor Kolom

        • ORDER BY juga dapat memakai nomor posisi kolom pada SELECT.
        • Contoh:
          SELECT title, pages
FROM book
ORDER BY 2;

          Penjelasan:

          • Kolom 1 → title
          • Kolom 2 → pages

          Jadi ORDER BY 2 = urut berdasarkan pages.

          Catatan:

          Teknik ini cepat ditulis, tetapi tidak disarankan untuk produksi karena membingungkan ketika daftar kolom SELECT berubah.

        Cara SQL Mengurutkan Teks

        SQL mengurutkan teks karakter per karakter:

        • Bandingkan huruf pertama.
        • Jika sama, lanjut ke huruf kedua.
        • Jika tiga kata pertama sama, SQL mulai membedakan di karakter berikutnya.

        Contoh urutan alfabet:

        • "Book of AI"
        • "Book of Data"
        • "Book of Machine Learning"

        Perilaku ORDER BY dengan NULL

        Secara umum:

        DatabaseNULL diurutkan (ASC)NULL diurutkan (DESC)
        SQL ServerAtas duluBawah
        MySQLAtas duluBawah
        PostgreSQLBawahAtas

        Beberapa DBMS mendukung:

        ORDER BY kolom NULLS FIRST;
ORDER BY kolom NULLS LAST;

        Contoh:

        ORDER BY pages ASC NULLS LAST;

Function & Grouping
  • Built-in Function

    Banyak database menyediakan built-in functions yang bisa kamu gunakan langsung di dalam query SQL. Fungsi ini membantu kamu melakukan perhitungan atau operasi lain langsung di dalam database, tanpa harus mengambil datanya dulu ke aplikasi atau notebook.

    Keuntungan menggunakan fungsi bawaan:

    • Mengurangi jumlah data yang perlu diambil.
    • Mengurangi beban jaringan.
    • Lebih cepat untuk kumpulan data yang besar.

    Tabel Contoh: PETRESCUE

    Tabel PETRESCUE berisi informasi penyelamatan hewan, dengan kolom:

    • ID
    • ANIMAL
    • QUANTITY
    • COST
    • RESCUEDATE

    Fungsi

    • Aggregate Functions

      Fungsi agregat bekerja pada sekumpulan nilai dalam kolom dan mengembalikan satu nilai saja.

      Jenis-jenis fungsi agregat:

      • SUM() – Menjumlahkan nilai dalam kolom
      • MIN() – Mengambil nilai terkecil
      • MAX() – Mengambil nilai terbesar
      • AVG() – Menghitung nilai rata-rata
      • COUNT() – Menghitung jumlah baris

      Contoh

      • Menjumlahkan nilai dalam kolom
        SELECT SUM(COST) FROM PETRESCUE;

        Memberikan total COST dari semua penyelamatan.

      • Memberi nama pada kolom hasil (alias)
        SELECT SUM(COST) AS SUM_OF_COST FROM PETRESCUE;
      • Nilai maksimum
        SELECT MAX(QUANTITY) FROM PETRESCUE;
      • Menerapkan fungsi agregat pada subset data
        SELECT MIN(ID) FROM PETRESCUE
WHERE ANIMAL = 'dog';
      • Rata-rata (Average)
        SELECT AVG(COST) FROM PETRESCUE;
      • Operasi antar kolom + fungsi agregat

        Menghitung biaya rata-rata per anjing:

        SELECT AVG(COST / QUANTITY)
FROM PETRESCUE
WHERE ANIMAL = 'dog';

    • Fungsi Skalar (Scalar Functions)

      Fungsi skalar bekerja pada satu nilai, bukan pada satu kolom penuh.

      Beberapa contoh:

      • ROUND() – membulatkan angka
      • LENGTH() – menghitung panjang string
      • UPPER() – mengubah huruf menjadi huruf besar
      • LOWER() – mengubah huruf menjadi huruf kecil

      Contoh

      • Membulatkan nilai
        SELECT ROUND(COST) FROM PETRESCUE;
      • Panjang string
        SELECT LENGTH(ANIMAL) FROM PETRESCUE;
      • Huruf besar
        SELECT UPPER(ANIMAL) FROM PETRESCUE;
      • Huruf kecil di klausa WHERE

        Berguna jika tidak tahu apakah data disimpan dalam uppercase/lowercase.

        SELECT *
FROM PETRESCUE
WHERE LOWER(ANIMAL) = 'cat';

    • Tambahan
      • COALESCE() — Ganti NULL dengan nilai tertentu

        Sangat sering dipakai dalam data cleaning.

        SELECT COALESCE(phone, 'Tidak ada') AS phone_clean
FROM customers;

      • IFNULL(), NVL(), ISNULL() — Alternatif COALESCE

        Bergantung pada jenis database, fungsinya sama: mengganti NULL.

        SELECT IFNULL(price, 0) FROM products;

      • CASE WHEN — Membuat kolom kondisi (IF versi SQL)

        Ini salah satu fungsi paling penting di SQL.

        SELECT
  cost,
  CASE
    WHEN cost > 50 THEN 'MAHAL'
    WHEN cost > 20 THEN 'SEDANG'
    ELSE 'MURAH'
  END AS kategori_harga
FROM petrescue;

      • CEIL() → pembulatan ke atas
        SELECT CEILING(10.2); -- hasilnya 11

      • FLOOR() → pembulatan ke bawah
        SELECT FLOOR(10.8); -- hasilnya 10

      • SUBSTRING() — Ngambil sebagian teks

        Contoh 1 - Mengambil 3 huruf pertama

        SELECT SUBSTRING(animal, 1, 3)
FROM petrescue;

        Contoh 2 - Mengekstrak nama depan dan nama akhir dari nama lengkap

        • Nama Depan
          SUBSTRING(full_name, 1, POSITION(' ' IN full_name) - 1)
        • Nama Belakang
          SUBSTRING(full_name, POSITION(' ' IN full_name) + 1)

      • REPLACE() — Mengganti teks tertentu
        SELECT REPLACE(animal, 'dog', 'puppy')
FROM petrescue;

      • CONCAT() — Menggabungkan teks
        SELECT CONCAT(first_name, ' ', last_name) AS full_name
FROM authors;

      • TRIM(), LTRIM(), RTRIM() — Membersihkan spasi

        Biasanya dipakai saat melakukan data cleaning.

        SELECT TRIM(animal) FROM petrescue;

      • CAST() / CONVERT() — Mengubah tipe data

        Ini sangat penting untuk analisis dan join.

        SELECT CAST(cost AS INT) FROM petrescue;

  • Grouping Result Sets

    Mengapa Perlu Mengelompokkan atau Menghilangkan Duplikasi?

    • Pada beberapa kasus, hasil dari SELECT dapat berisi nilai-nilai yang berulang.
    • Contoh: kolom country pada tabel authors yang memiliki banyak penulis dari negara yang sama.
    • Jika kita hanya ingin:
      • Daftar negara unik
        atau
      • Jumlah penulis per negara

      maka kita perlu teknik pengelompokan.

    Jenis

    • Menghilangkan Duplikat dengan DISTINCT
      • Ketika SELECT menghasilkan nilai berulang:
        SELECT country
FROM authors
ORDER BY country;

        Karena beberapa penulis berasal dari negara yang sama, hasilnya memuat duplikat.

      • Menghapus duplikat:
        SELECT DISTINCT country
FROM authors
ORDER BY country;

        DISTINCT membuat hasil menjadi unik, misalnya dari 20 baris → hanya 6 negara.

      • Catatan tambahan:

        DISTINCT berlaku pada kombinasi seluruh kolom yang ditulis di SELECT.
        Misal:

        SELECT DISTINCT country, city

        → unik berdasarkan pasangan (country, city).

    • Mengelompokkan Data dengan GROUP BY

      Jika selain daftar negara unik kita juga ingin menghitung jumlah penulis tiap negara, gunakan GROUP BY.

      • Contoh dasar:
        SELECT country, COUNT(*)
FROM authors
GROUP BY country;

        GROUP BY mengelompokkan semua baris berdasarkan nilai country, lalu COUNT(*) menghitung jumlah baris di setiap kelompok.

      • Masalah penamaan kolom (alias)

        Jika tidak diberi alias, kolom hasil fungsi agregat akan muncul sebagai angka (misal “2” atau “count”).

        Untuk itu gunakan AS:

        SELECT country, COUNT(*) AS jumlah_penulis
FROM authors
GROUP BY country;

        Alias membuat hasil lebih jelas dibaca.

    • Membatasi Hasil Pengelompokan dengan HAVING
      • Jika ingin menerapkan kondisi setelah dikelompokkan, gunakan HAVING.
      • Contoh kebutuhan:

        Tampilkan hanya negara dengan lebih dari 4 penulis.

        Kode:
SELECT country, COUNT(*) AS jumlah_penulis
FROM authors
GROUP BY country
HAVING COUNT(*) > 4;

        Hasilnya hanya negara yang jumlah penulisnya ≥ 5.
        Misalnya:

        • China → 6 penulis
        • India → 6 penulis
      • Perbedaan WHERE vs HAVING
        KlausaDigunakan UntukWaktu Eksekusi
        WHEREMenyaring baris sebelum dikelompokkanSebelum GROUP BY
        HAVINGMenyaring setelah pengelompokanSetelah GROUP BY

    Urutan Eksekusi Penting

    Agar tidak bingung, berikut urutan SQL sebenarnya berjalan:

    • FROM
    • WHERE
    • GROUP BY
    • HAVING
    • SELECT
    • ORDER BY

    Maka:

    • WHERE tidak bisa memakai fungsi agregat (COUNT, SUM, dll).
    • HAVING diperbolehkan memakai fungsi agregat.

JOIN & Relational Thinking

Konsep

JOIN digunakan untuk menggabungkan dua tabel berdasarkan sebuah kolom yang saling berhubungan (biasanya PK–FK).

  • Tabel A = biasanya tabel utama (misal Employee)
  • Tabel B = tabel kedua tambahan (misal Department)

JOIN memastikan data yang berhubungan berdampingan dalam 1 baris hasil query.

Jenis

  • INNER JOIN
    • Konsep

      INNER JOIN hanya mengembalikan baris yang matching di kedua tabel.

      • Kalau baris A punya pasangan di tabel B → diambil
      • Kalau tidak punya pasangan → dibuang

    • Intuisi
      • Tabel A = daftar orang yang bayar seminar
      • Tabel B = daftar orang yang hadir
      INNER JOIN = yang bayar dan hadir.

      Kalau cuma bayar tapi gak hadir → hilang.

      Cuma hadir tapi gak bayar → hilang.

      A ●●●●
   ●●●● B

      • Filter data yang “valid”
      • Menghindari row yang tidak punya foreign key
      • Membuat dataset bersih

    • Kesalahan
      • Salah taruh kondisi join di WHERE → hasil bisa lebih sedikit
      • Lupa ON → berubah jadi CROSS JOIN
      • JOIN kolom yang salah → hasil meledak (duplicate)

      • Tabel kiri = prioritas utama
      • Tabel kanan = hanya tambahan
      • Meskipun tidak ada pasangan → tetap tampil.

      • Mencari siapa yang tidak punya pasangan

Subqueries
  • Introduction
    • Subquery = query di dalam query.

      Gampangnya begini:

      Kamu membuat satu query kecil (anak), lalu hasilnya dipakai oleh query utama (induk).
    • Subquery biasanya ditulis di dalam:
      • SELECT
      • FROM
      • WHERE
      • HAVING
    • Fungsi utamanya:
      • Mengambil nilai tambahan tanpa membuat tabel baru
      • Membandingkan data dengan hasil perhitungan lain
      • Membuat filtering yang lebih pintar
    • Ilustrasi sederhana:
      SELECT full_name,
       salary,
       (SELECT AVG(salary) FROM Employee)
FROM Employee;

      Subquery tersebut menghitung rata-rata gaji lalu dikembalikan ke query utama.

  • Type by Position
    • Subquery at SELECT (Scalar Subquery)
      • Konsep
        • Subquery ini mengembalikan satu nilai saja (1 baris, 1 kolom).
        • Karena SELECT membutuhkan kolom, maka subquery di sini hanya boleh mengembalikan 1 nilai. Kalau hasilnya lebih dari 1, SQL akan error.
        • Biasanya dipakai untuk:
          • menambah kolom perhitungan
          • menghitung statistik global
          • menghitung statistik per grup dengan correlated subquery

        Sifat / Aturan

        • HARUS mengembalikan satu nilai → disebut scalar subquery.
        • Boleh bersifat correlated, artinya membaca kolom dari query luar.
        • Bisa muncul lebih dari satu subquery dalam SELECT.

      • Contoh

        Menambah kolom rata-rata gaji perusahaan

        SELECT
  full_name,
  salary,
  (SELECT AVG(salary) FROM Employee) AS avg_salary_per_company
FROM Employee;

        Keterangan:

        • Subquery menghitung rata-rata gaji secara global.
        • Semua baris akan memiliki nilai kolom tambahan yang sama.

        Rata-rata gaji per departemen (correlated)

        SELECT
  e.full_name,
  e.salary,
  (SELECT AVG(salary)
FROM Employee e2
WHERE e2.dept_id = e.dept_id) AS avg_salary_per_dept
FROM Employee e;

        Keterangan:

        • Subquery dibaca per baris.
        • Untuk karyawan di dept_id 2 → subquery menghitung rata-rata gaji dept 2.

        Hitung jumlah proyek per karyawan

        SELECT
    e.full_name,
    (SELECT COUNT(*) FROM Project p WHERE p.emp_id = e.emp_id) AS total_projects
FROM Employee e;

        Keterangan:

        • Subquery membaca emp_id dari query luar.
        • Memberi kolom tambahan jumlah proyek.

    • Subquery at WHERE

      Konsep

      • Subquery yang dipakai untuk filtering data.
      • WHERE dipakai untuk memilih baris tertentu, sehingga subquery di sini harus menghasilkan nilai yang bisa dipakai untuk:
        • perbandingan tunggal (=, >, <, >=)
        • list (IN)
        • kondisi kompleks (ANY, ALL, EXISTS)

      Sifat / Aturan

      • Hasil subquery boleh satu nilai atau banyak nilai → tergantung operator.
      • Bisa correlated atau tidak.
      • Paling sering dipakai daripada jenis subquery lainnya.

      Contoh

      • Gaji di atas rata-rata perusahaan
        SELECT full_name, salary
FROM Employee
WHERE salary > (
    SELECT AVG(salary) FROM Employee
);
      • Filtering berdasarkan daftar departemen
        SELECT full_name
FROM Employee
WHERE dept_id IN (
    SELECT dept_id
    FROM Department
    WHERE location = 'Jakarta'
);

        Keterangan:

        • Subquery menghasilkan banyak dept_id.
        • Query luar memilih karyawan yang dept_id-nya ada di list itu.
      • Correlated subquery di WHERE

        Ambil karyawan yang gajinya di atas rata-rata departemennya sendiri.

        SELECT full_name, salary
FROM Employee e
WHERE salary > (
    SELECT AVG(salary)
    FROM Employee e2
    WHERE e2.dept_id = e.dept_id
);
      • Gunakan EXISTS (lebih efisien)
        SELECT full_name
FROM Employee e
WHERE EXISTS (
    SELECT 1
    FROM Project p
    WHERE p.emp_id = e.emp_id
);

        Keterangan:

        • Pilih karyawan yang punya proyek.
        • EXISTS = hanya mengecek keberadaan, bukan nilai.
    • Subquery at HAVING

      Konsep

      • HAVING bekerja setelah GROUP BY, dan biasanya dipakai untuk memfilter grup, bukan baris individu.
      • Subquery di HAVING digunakan untuk:
        • membandingkan hasil agregasi per grup dengan agregasi lain
        • filtering berdasarkan perhitungan di luar grup

      Sifat / Aturan

      • Biasanya digunakan dengan GROUP BY.
      • Hasil subquery biasanya scalar atau aggregate.

      Contoh

      • Ambil departemen yang total gajinya di atas rata-rata total gaji semua departemen
        SELECT dept_id, SUM(salary) AS total_salary
FROM Employee
GROUP BY dept_id
HAVING SUM(salary) > (
    SELECT AVG(total_salary)
    FROM (
        SELECT SUM(salary) AS total_salary
        FROM Employee
        GROUP BY dept_id
    ) AS x
);

        Keterangan:

        Subquery dalam HAVING membandingkan total gaji per departemen dengan rata-rata total gaji antar departemen.

      • Departemen yang memiliki jumlah karyawan lebih banyak dari rata-rata perusahaan
        SELECT dept_id, COUNT(*) AS emp_count
FROM Employee
GROUP BY dept_id
HAVING COUNT(*) > (
    SELECT AVG(cnt)
    FROM (
        SELECT COUNT(*) AS cnt
        FROM Employee
        GROUP BY dept_id
    ) AS z
);
    • Subquery at FROM (Derived Table / Inline View)

      Konsep

      • Subquery di FROM menghasilkan tabel sementara yang bisa diperlakukan seperti tabel normal.
      • Subquery ini sering disebut:
        • Derived table
        • Inline view
        • Table subquery
      • Biasanya digunakan ketika:
        • GROUP BY perlu diproses lagi
        • ingin membersihkan query yang kompleks
        • ingin membuat “tabel sementara” tanpa membuat tabel fisik

      Sifat / Aturan

      • Wajib punya alias (AS nama_tabel).
      • Bisa mengembalikan banyak baris dan banyak kolom.
      • Bisa berisi GROUP BY, HAVING, JOIN, dll.

      Contoh

      • Hitung total gaji per departemen, lalu pilih dept yang totalnya > 100M
        SELECT *
FROM (
    SELECT dept_id, SUM(salary) AS total_salary
    FROM Employee
    GROUP BY dept_id
) AS t
WHERE total_salary > 100000000;

        Keterangan:

        • Subquery menghasilkan tabel berisi total gaji per dept.
        • Query luar hanya melakukan filtering.
      • Hitung jumlah proyek per karyawan lalu ambil yang paling aktif
        SELECT emp_id, total_projects
FROM (
    SELECT emp_id, COUNT(*) AS total_projects
    FROM Project
    GROUP BY emp_id
) AS p
ORDER BY total_projects DESC;
      • Derived table + join
        SELECT e.full_name, t.total_projects
FROM Employee e
JOIN (
    SELECT emp_id, COUNT(*) AS total_projects
    FROM Project
    GROUP BY emp_id
) AS t
ON e.emp_id = t.emp_id;

        Keterangan:

        • Derived table t berisi jumlah proyek per karyawan.
        • Lalu di-join dengan tabel Employee.

    Subquery dapat mengembalikan:

    • 1 nilai (1 baris, 1 kolom) → Scalar Subquery
    • 1 baris, banyak kolom → Single-Row Subquery
    • Banyak baris (list) → Multi-Row Subquery
    • Banyak baris + tergantung baris luar → Correlated Subquery (kombinasi khusus)

      Konsep

      • Scalar = skala kecil → subquery ini mengembalikan hanya 1 nilai.
      • Contoh hasilnya:

"Jakarta"
      • Karena cuma 1 nilai, scalar subquery bisa dipakai di mana saja nilai tunggal diperlukan, seperti:
        • di SELECT

        • di ORDER BY

      Ciri-ciri

      • Output selalu 1 baris 1 kolom
      • Tidak boleh mengembalikan lebih dari 1 baris → kalau lebih, SQL ERROR

      Contoh

      • Rata-rata gaji perusahaan

    full_name,
    salary,
    (SELECT AVG(salary) FROM Employee) AS avg_salary
FROM Employee;

        Subquery → AVG(salary) → 1 nilai.

FROM Employee
WHERE salary > (SELECT AVG(salary) FROM Employee);

        (SELECT AVG...) = scalar.

      • Hitung total karyawan
        SELECT
    dept_name,

      Konsep

      • Mengembalikan 1 baris, tetapi bisa lebih dari satu kolom.

      • Operatornya bukan IN, tapi (col1, col2) = (...)

      $ $

FROM Employee
WHERE (dept_id, salary) = (

CTE
  • Introduction

    Definisi

    • Common Table Expression (CTE) adalah tabel sementara yang hanya berlaku selama satu pernyataan SQL dieksekusi.
    • CTE dideklarasikan menggunakan kata kunci:
      WITH nama_cte AS (
    query
)
SELECT ...
    • Secara formal:
      CTE digunakan untuk membuat hasil query perantara yang bisa dipanggil kembali di query utama, sehingga meningkatkan keterbacaan dan modularitas SQL.

      Intuitifnya:

      CTE itu kayak membuat “ruang kerja sementara”. Kamu ambil data dulu, beri nama, lalu pakai ulang dengan mudah.

    Kenapa CTE Dibutuhkan?

    • Secara formal:
      • Mengurangi kompleksitas query.
      • Mempermudah debugging.
      • Meningkatkan readability.
      • Menghindari subquery yang berulang.
      • Memfasilitasi pembuatan query bertingkat (multi-step query).
    • Secara intuitif:
      Tanpa CTE, query panjang jadi berantakan seperti kabel kusut. Dengan CTE, semua jadi rapi step-by-step.


    CTE Hanya Hidup Selama Satu Query

    • Formal:

      CTE tidak disimpan sebagai tabel fisik dan tidak bertahan setelah query selesai dieksekusi.

    • Intuitif:
      “Tabel sementara yang langsung hilang begitu SELECT selesai.”

      Bukan VIEW, bukan TEMP TABLE, bukan tabel permanen.

    Struktur Dasar CTE

    • Formal:
      WITH cte_name(column1, column2, ...)
AS (
    SELECT ...
)
SELECT ...
FROM cte_name;
    • Intuitif:
      • Buat kotak kerja → beri nama → isi datanya
      • Gunakan kotak itu

    Contoh Paling Sederhana

    • Tanpa CTE (subquery di FROM)
      SELECT *
FROM (
    SELECT dept_id, COUNT(*) AS total_employee
    FROM employee
    GROUP BY dept_id
) AS x;
    • Dengan CTE (lebih formal dan bersih)
      WITH dept_summary AS (
    SELECT dept_id, COUNT(*) AS total_employee
    FROM employee
    GROUP BY dept_id
)
SELECT *
FROM dept_summary;

      Intuitif:

      “Buat dulu tabel ringkasan department, lalu tampilkan.”

    Kapan Menggunakan CTE?

    • Formal:
      • Ketika query melibatkan banyak operasi transformasi.
      • Ketika subquery harus dipakai berulang.
      • Ketika readability penting.
      • Ketika query memiliki langkah analisis bertahap.
    • Intuitif:
      Gunakan CTE saat kamu mulai merasa “query ini mulai panjang dan bikin pusing”.

    Limitasi CTE

    • Formal:
      • Tidak otomatis membuat index.
      • Tidak disimpan di disk.
      • Tidak bisa dipakai di query berbeda.
    • Intuitif:
      CTE itu sekali pakai. Begitu SELECT selesai → hilang.

  • With JOIN and Aggregation

    Konsep

    CTE (Common Table Expression) dapat menyimpan hasil sementara yang kemudian bisa digabungkan (JOIN) dengan tabel lain.

    Ini sangat berguna saat:

    • Kamu perlu membuat hasil antara yang lebih rapi sebelum digabungkan.
    • Kamu ingin menghindari nested subquery yang berulang.
    • Kamu ingin membangun query dari langkah-langkah logis yang mudah dibaca.

    Secara formal:

    CTE dapat digunakan sebagai "tabel virtual" untuk proses JOIN, agregasi lanjutan, dan filtering, sebelum menampilkan hasil akhirnya.

    Case 1 - CTE + JOIN

    • Deskripsi

      Jumlah karyawan per departemen lalu menampilkan nama departemennya.

    • Query
      • CTE
        WITH employee_count AS (
    SELECT dept_id, COUNT(*) AS total_employee
    FROM employee
    GROUP BY dept_id
)
      • Main Query
        SELECT d.dept_name, ec.total_employee
FROM department d
LEFT JOIN employee_count ec
       ON d.dept_id = ec.dept_id;

        Keterangan

        • CTE bikin daftar “berapa jumlah karyawan per dept”.
        • Main query menggabungkan daftar itu dengan tabel department.
          Ibaratnya kamu bikin tabel kecil dulu (“jumlah orang per ruangan”), baru kamu tempelkan ke “nama ruangan”.

    Case 2 - CTE untuk Membersihkan Step

    • Deskripsi

      Hitung total jam kerja karyawan, lalu ambil hanya yang jam kerjanya > 40.

    • Query
      • CTE
        WITH work_hours AS (
    SELECT emp_id, SUM(hours) AS total_hours
    FROM project
    GROUP BY emp_id
)
      • Main Query
        SELECT e.full_name, w.total_hours
FROM employee e
JOIN work_hours w ON e.emp_id = w.emp_id
WHERE w.total_hours > 40;

        Keterangan:

        Kamu tidak hanya membuat CTE, tapi kamu:

        • melakukan agregasi,
        • lalu JOIN ke tabel lain,
        • lalu filtering di luar CTE.

    Case 3 - CTE + Multiple JOIN

    • Deskripsi

      total jam kerja per department.

    • Query
      • CTE
        WITH emp_hours AS (
    SELECT emp_id, SUM(hours) AS total_hours
    FROM project
    GROUP BY emp_id
)
      • Main Query
        SELECT d.dept_name, SUM(eh.total_hours) AS dept_total_hours
FROM emp_hours eh
JOIN employee e ON eh.emp_id = e.emp_id
JOIN department d ON e.dept_id = d.dept_id
GROUP BY d.dept_name;

        Keterangan:

        • CTE ngehitung total jam kerja tiap karyawan.
        • Lalu "jam kerja" itu ditempel ke employee → department.
        • Baru dihitung ulang per department.

        Mirip bikin tabel kecil dulu baru dihitung ulang di konteks lebih besar.

  • Multi-CTE

    Konsep

    Multi-CTE adalah penggunaan lebih dari satu CTE dalam satu query, di mana:

    • Tiap CTE bisa berdiri sendiri,
    • Atau satu CTE bisa menggunakan hasil dari CTE sebelumnya,
    • Dan semua CTE berada sebelum main query,
    • Dipisahkan dengan koma.

    Struktur formal:

    WITH cte1 AS (...),
     cte2 AS (...),
     cte3 AS (...)
SELECT ...
FROM ...

    Secara formal, ini membantu membangun query secara modular — memecah perhitungan kompleks menjadi blok kecil yang jelas.

    Intuisi

    Bayangkan seperti bikin proses:

    • CTE 1: Bikin bahan mentah
    • CTE 2: Olah bahan 1 jadi bahan setengah jadi
    • CTE 3: Olah lagi jadi bahan matang
    • Main query: Sajikan hasil akhir

    Jadi multi-CTE itu kayak bikin "step-step kecil" yang gampang dibaca, dibanding satu query raksasa yang bikin pusing.

    Contoh 1 - Hitung total jam per employee → total jam per dept → kategori departemen

    • CTE 1 — Total jam kerja per employee
      WITH emp_hours AS (
    SELECT emp_id, SUM(hours) AS total_hours
    FROM project
    GROUP BY emp_id
),
    • CTE 2 — Total jam kerja per department
      dept_hours AS (
    SELECT e.dept_id, SUM(eh.total_hours) AS dept_total_hours
    FROM emp_hours eh
    JOIN employee e ON eh.emp_id = e.emp_id
    GROUP BY e.dept_id
),
    • CTE 3 — Klasifikasi departemen
      dept_category AS (
    SELECT
        dept_id,
        dept_total_hours,
        CASE
            WHEN dept_total_hours >= 200 THEN 'High Workload'
            WHEN dept_total_hours >= 100 THEN 'Medium Workload'
            ELSE 'Low Workload'
        END AS workload_level
    FROM dept_hours
)

    Contoh 2- Multi-CTE untuk filtering bertahap

    • Ambil karyawan dengan gaji > median (anggap kita sudah hitung median manual)
      WITH high_salary AS (
    SELECT *
    FROM employee
    WHERE salary > 5000
),
    • Ambil jam kerjanya
      workload AS (
    SELECT h.emp_id, SUM(p.hours) AS total_hours
    FROM high_salary h
    JOIN project p ON h.emp_id = p.emp_id
    GROUP BY h.emp_id
    )
    • Main Query
      SELECT e.full_name, w.total_hours
FROM workload w
JOIN employee e ON w.emp_id = e.emp_id;

      Intuisi:

      • Pilih dulu karyawan high salary → baru hitung jam kerja mereka.
      • Kalau kamu gabung langsung, query jadi panjang dan susah dibaca.
      • Multi-CTE sebagai Data Pipeline Mini
        • Bersihkan data project yang statusnya
          WITH active_project AS (
    SELECT *
    FROM project
    WHERE status = 'active'
),
        • Hitung total jam aktif per karyawan
          emp_active_hours AS (
    SELECT emp_id, SUM(hours) AS active_hours
    FROM active_project
    GROUP BY emp_id
),
        • Label effort
          effort_level AS (
    SELECT
        emp_id,
        active_hours,
        CASE
            WHEN active_hours > 50 THEN 'Top Contributor'
            ELSE 'Normal'
        END AS level
    FROM emp_active_hours
)
        • Main Query
          SELECT e.full_name, l.level, l.active_hours
FROM effort_level l
JOIN employee e ON l.emp_id = e.emp_id;

Window Functions
  • Konsep

    Sebelum window function ada, SQL punya keterbatasan besar.

    SQL klasik hanya punya 2 mode:

    1. Detail per baris
    1. Agregasi (GROUP BY)

    ❌ Tapi tidak bisa di tengah-tengah

    Contoh masalah nyata

    Tabel sales:

    datecustomeramount
    01-01A100
    01-01B150
    01-02A200

    ❓ Pertanyaan:

    “Tampilkan setiap transaksi, tapi juga tampilkan total belanja customer di setiap baris.”

    Yang bisa dilakukan SQL lama

    • Detail (tanpa agregasi)
      SELECT * FROM sales;

      👉 Dapat detail, tapi tidak tahu total customer

    • GROUP BY
      SELECT customer, SUM(amount)
FROM sales
GROUP BY customer;

      👉 Dapat total, tapi detail transaksi hilang

    📌 GAP INI yang melahirkan Window Function

    Intuisi

    Window Function = agregasi TANPA menghancurkan baris

    Kalimat kunci:

    “Hitung sesuatu di atas sekumpulan baris,
    tapi hasilnya ditempel ke setiap baris.”

    Visual Mental

    Bayangkan:

    • Data = barisan orang
    • Window = kaca transparan yang bisa kamu geser
    • Kamu berdiri di satu orang, tapi bisa melihat beberapa orang lain

    Apa Itu “WINDOW”?

    Window BUKAN tabel baru
    Window BUKAN hasil join
    Window = konteks pandang sementara

    Untuk setiap baris, SQL:

    1. Menentukan baris mana yang relevan
    1. Menghitung fungsi
    1. Menaruh hasil ke baris tersebut

    Komponen

    Bukan sintaks dulu — konsepnya dulu.

    • Current Row (Baris Aktif)

      SQL selalu berpikir:

      “Sekarang aku berdiri di baris ini”

      Semua perhitungan relatif terhadap baris ini

    • Partition (Kelompok Logis)
      “Baris ini boleh melihat siapa saja?”

      📌 PARTITION BY menentukan:

      • batas “dunia” baris itu
      • kapan perhitungan reset

      Analogi:

      • Satu kelas → satu partition
      • Tidak boleh lihat kelas lain
    • Order (Arah Pandang)
      “Urutan barisnya gimana?”

      Tanpa urutan:

      • Tidak ada “sebelumnya”
      • Tidak ada “berikutnya”
      • Tidak ada “running”

      📌 ORDER BY memberi arah waktu / ranking

    • Frame (Jarak Pandang)
      “Sejauh apa aku boleh melihat?”

      Misalnya:

      • cuma 3 baris ke belakang
      • dari awal sampai sekarang
      • cuma baris saat ini

      Ini konsep window frame

    WINDOW FUNCTION vs GROUP BY (SECARA FILOSOFI)

    GROUP BYWINDOW FUNCTION
    Merangkum dataMengkontekstualkan data
    Mengurangi barisMenjaga detail
    Final outputLayer analisis
    Jawaban statisInsight per baris

    📌 Data Analyst lebih sering butuh konteks, bukan cuma ringkasan.

    KENAPA WINDOW FUNCTION BUKAN FOR LOOP

    Secara cara mikir, mirip.
    Secara eksekusi, beda total.

    For Loop

    • Baris diproses satu per satu
    • State disimpan manual

    Window Function

    • SQL engine memandang seluruh set
    • Optimizer menentukan cara tercepat
    • Bisa paralel
    • Tidak ada state manual

    📌 Kamu tidak mengontrol “bagaimana”, cuma “apa”

    Urutan Evaluasi (KONSEPTUAL)

    Secara mental (bukan teknis mesin):

    1. FROM
    1. WHERE
    1. GROUP BY
    1. SELECT
    1. WINDOW FUNCTION DIEVALUASI
    1. ORDER BY (akhir)

    📌 Artinya:

    • Window function melihat hasil filter
    • Tapi sebelum output akhir

    Tipe Masalah yang “TERIAK” MINTA WINDOW FUNCTION

    Kalau kamu melihat kata-kata ini di soal:

    • “per …”
    • “dibandingkan dengan sebelumnya”
    • “running”
    • “ranking”
    • “kontribusi”
    • “top N per kategori”
    • “growth”

    🚨 90% → window function

    Kesalahan Konseptual Umum

    ❌ Mengira window function mengubah jumlah baris
    ❌ Mengira window = tabel baru
    ❌ Mengira hasil window bisa langsung difilter di WHERE
    ❌ Mengira semua agregasi perlu GROUP BY

    Rangkuman

    Window function memberi KACAMATA analisis
    pada setiap baris data.

    Sintaks Umum

    fungsi() OVER (
    PARTITION BY kolom
    ORDER BY kolom
    ROWS / RANGE
)

    Komponen Penting

    KomponenFungsi
    OVER()Menandakan window function
    PARTITION BYMembagi data jadi kelompok
    ORDER BYUrutan perhitungan
    ROWS / RANGEBatas window (opsional)
  • PARTITION BY — Membagi Data

    Masalah yang Diselesaikan PARTITION BY

    • Bayangin kamu punya data ini:

      Tabel: sales

      datecategoryrevenue
      01-01A100
      01-02A150
      01-01B200
      01-02B300

      ❓ Pertanyaan bisnis:

      “Berapa total revenue per category,
      tapi tetap tampilkan per hari?”
    • ❌ GROUP BY (gagal untuk kebutuhan ini)
      SELECT
    category,
    SUM(revenue)
FROM sales
GROUP BY category;

      ➡️ Detail tanggal hilang

      📌 Di sinilah PARTITION BY muncul.

    Definisi PARTITION BY

    PARTITION BY membagi baris data menjadi kelompok logis (partition),
    lalu window function dihitung TERPISAH di setiap kelompok,
    TANPA menggabungkan baris.

    Kalimat sederhananya:

    “Hitung per kelompok, tapi tampilkan di setiap baris.”

    Mental Model (INI PENTING)

    Bayangin:

    • Dataset = satu ruangan besar
    • PARTITION BY category = tembok kaca antar kategori
    • Baris hanya bisa melihat baris lain di dalam tembok yang sama

    📌 Tidak ada interaksi antar partition.

    PARTITION BY = GROUP BY versi “tidak menghancurkan data”

    GROUP BYPARTITION BY
    Menggabungkan barisMempertahankan baris
    Output lebih sedikitOutput tetap
    Summary tableEnriched table

    Contoh DASAR (Tanpa ORDER Dulu)

    SELECT
    date,
    category,
    revenue,
    SUM(revenue) OVER (PARTITION BY category) AS total_category
FROM sales;

    PROSES KONSEPTUAL

    • PARTITION A
      daterevenue
      01-01100
      01-02150

      ➡️ total = 250

    • PARTITION B
      daterevenue
      01-01200
      01-02300

      ➡️ total = 500

    • HASIL AKHIR
      datecategoryrevenuetotal_category
      01-01A100250
      01-02A150250
      01-01B200500
      01-02B300500

      📌 Total yang sama diulang di setiap baris dalam partition

    PARTITION BY + ORDER BY (BARU ADA “URUTAN”)

    Pertanyaan baru:

    “Total kumulatif per category per hari?”
    SELECT
    date,
    category,
    revenue,
    SUM(revenue) OVER (
        PARTITION BY category
        ORDER BY date
    ) AS running_category
FROM sales;

    🔍 PROSES

    • PARTITION A (urut tanggal)
      daterevenuerunning
      01-01100100
      01-02150250
    • PARTITION B
      daterevenuerunning
      01-01200200
      01-02300500
    • HASIL
      datecategoryrevenuerunning_category
      01-01A100100
      01-02A150250
      01-01B200200
      01-02B300500

      📌 Urutan di-reset di setiap partition

    Multiple PARTITION Columns

    PARTITION BY category, region

    📌 Artinya:

    • Category A di Region X ≠ Category A di Region Y
    • Kombinasi kolom = identity partition

    PARTITION BY TANPA ORDER BY — KAPAN DIPAKAI?

    Dipakai kalau:

    • Mau total / average / count per grup
    • Tidak peduli urutan

    Contoh:

    • total belanja customer
    • rata-rata gaji departemen

    Kesalahan Pemula yang Umum

    ❌ Mengira PARTITION BY mengurutkan data
    ❌ Mengira PARTITION BY wajib ada
    ❌ Mengira sama persis dengan GROUP BY
    ❌ Bingung kenapa nilai diulang

    • --Kalimat EMAS (WAJIB DIINGAT)
    PARTITION BY menentukan “dunia” setiap baris.
    Dunia satu baris tidak pernah bercampur dengan dunia lain.

    Contoh lain

    SELECT
    department,
    employee,
    salary,
    AVG(salary) OVER (PARTITION BY department) AS avg_dept_salary
FROM employees;

    Interpretasi

    • Hitung rata-rata gaji per department
    • Nilai rata-rata muncul di setiap baris employee

  • ORDER BY di Window Function

    Masalah yang Diselesaikan ORDER BY

    Tanpa ORDER BY, window function tidak tahu arah.

    Artinya:

    • tidak ada “sebelumnya”
    • tidak ada “berikutnya”
    • tidak ada “kumulatif”
    • tidak ada “ranking”

    📌 Semua baris dianggap setara & tak berurutan.

    Definisi ORDER BY di Window

    ORDER BY di window function menentukan urutan logis
    di dalam setiap partition
    untuk menghitung nilai relatif antar baris.

    Bukan buat tampilan akhir, tapi buat cara menghitung.

    ORDER BY di Window ≠ ORDER BY di Akhir Query

    Ini kesalahan paling sering.

    • ORDER BY (akhir query)
      ORDER BY date;

      ➡️ Mengatur tampilan hasil

    • ORDER BY (di OVER)
      OVER (ORDER BY date)

      ➡️ Mengatur logika perhitungan

      📌 Dua hal yang berbeda total.

    Tanpa ORDER BY — Apa yang Terjadi?

    SUM(revenue) OVER ()

    SQL membaca:

    “Jumlahkan SEMUA baris, tempel ke setiap baris”
    daterevenuesum
    01-01100750
    01-02150750
    01-03120750

    📌 Tidak ada urutan → hasilnya sama semua

    ORDER BY → Mulai Ada “WAKTU” & “ARAH”

    SUM(revenue) OVER (ORDER BY date)

    Sekarang SQL berpikir:

    “Aku berdiri di baris ini, dan aku tahu mana yang duluan”
    • 🔍 PROSES KONSEPTUAL
      Baris aktifBaris yang dihitung
      01-0101-01
      01-0201-01 → 01-02
      01-0301-01 → 01-03
    • 🧾 HASIL
      daterevenuerunning_total
      01-01100100
      01-02150250
      01-03120370

    📌 ORDER BY = arah pandang ke depan / belakang

    ORDER BY + PARTITION BY (RESET LOGIKA)

    OVER (
    PARTITION BY category
    ORDER BY date
)

    Artinya:

    1. Pecah dunia berdasarkan category
    1. Urutkan waktu di dalam dunia itu
    1. Hitung masing-masing terpisah

    Visual Mental:

    Category A: 01-01 → 01-02 → 01-03
Category B: 01-01 → 01-02

    📌 Tidak ada cross-category influence.

    ORDER BY & RANKING (RELATIF ANTAR BARIS)

    • Tanpa ORDER BY:
      RANK() OVER ()

      ❌ Tidak masuk akal — mau ranking berdasarkan apa?

    • Dengan ORDER BY:
      RANK() OVER (ORDER BY revenue DESC)

      SQL berpikir:

      “Urutkan revenue dari besar ke kecil, lalu beri peringkat”

    ORDER BY + LAG / LEAD (BARU ADA SEBELUM / SESUDAH)

    LAG(revenue) OVER (ORDER BY date)

    SQL:

    “Ambil revenue baris SEBELUM ini, menurut urutan date”

    Tanpa ORDER BY → tidak ada konsep sebelumnya

    ORDER BY BISA LEBIH DARI SATU KOLOM

    ORDER BY year, month

    📌 Penting untuk:

    • data time series
    • tie-breaker
    • konsistensi urutan

    Kesalahan Umum Pemula

    ❌ Mengira ORDER BY di window mengurutkan output
    ❌ Mengira ORDER BY opsional untuk LAG/RANK
    ❌ Mengira urutan alami tabel itu pasti
    ❌ Tidak sadar hasil berubah kalau urutan berubah

    Rangkuman

    ORDER BY di window bukan soal tampilan,
    tapi soal logika perhitungan.

    Kapan HARUS pakai ORDER BY?

    Jika pertanyaan mengandung:

    • sebelumnya / sesudahnya
    • kumulatif
    • ranking
    • pertumbuhan
    • tren waktu

    🚨 ORDER BY WAJIB ADA

  • Aggregate Window Functions

    Fungsi yang Umum

    FungsiKegunaan
    SUM()Total
    AVG()Rata-rata
    MIN()Nilai minimum
    MAX()Nilai maksimum
    COUNT()Jumlah data

    Contoh: Total per Customer

    SELECT
    customer_id,
    order_id,
    amount,
    SUM(amount) OVER (PARTITION BY customer_id) AS total_customer_spend
FROM orders;

    Interpretasi

    • Total belanja customer dihitung
    • Tapi tetap tampil per order

    📌 Inilah keunggulan Window Function

  • Ranking Functions
    • Konsep

      Masalah ANALISIS yang Diselesaikan Ranking

      Data analyst sering ditanya:

      • Produk terlaris peringkat ke berapa?
      • Top customer per kategori?
      • Siapa yang terbaik / terburuk?
      • Urutan performa tanpa menghilangkan detail data

      📌 Masalah ini tidak bisa dijawab pakai GROUP BY saja.

      Apa itu Ranking Function?

      Ranking Function adalah window function
      yang memberi urutan (peringkat) ke setiap baris
      berdasarkan kriteria tertentu
      TANPA menggabungkan baris.

      Kata kunci:

      • Urutan
      • Relatif
      • Per baris
      • Butuh ORDER BY

      Syarat WAJIB Ranking Function

      🚨 Tanpa ini → ranking tidak masuk akal

      OVER (ORDER BY kolom)

      Karena:

      “Ranking itu selalu relatif terhadap urutan”

      Tiga Ranking Function UTAMA

      FungsiIntuisi Singkat
      ROW_NUMBER()Nomor unik, tidak peduli nilai sama
      RANK()Nilai sama → peringkat sama, tapi lompat
      DENSE_RANK()Nilai sama → peringkat sama, tidak lompat

      Sekarang kita bedah SATU PER SATU.

    • Fungsi

      Dataset Contoh (Pegangan Tetap)

      productsales
      A100
      B200
      C200
      D300

      Urutan logis (DESC):

      D → B → C → A

      Jenis

      • ROW_NUMBER() — Nomor Baris Unik

        Definisi

        Memberi nomor unik ke setiap baris
        berdasarkan urutan ORDER BY

        Query

        ROW_NUMBER() OVER (ORDER BY sales DESC)

        PROSES KONSEPTUAL

        UrutanProductSalesRow Number
        1D3001
        2B2002
        3C2003
        4A1004

        📌 Walaupun sales B & C sama, nomornya beda

        Kapan Dipakai?

        • Ambil Top N
        • Deduplication
        • Pagination

      • RANK() — Ranking Klasik (LOMPAT)

        Definisi

        Nilai sama → ranking sama
        Tapi ranking berikutnya melompat

        Query

        RANK() OVER (ORDER BY sales DESC)

        🔍 PROSES

        ProductSalesRank
        D3001
        B2002
        C2002
        A1004

        📌 Rank 3 tidak ada
        Karena ada dua item di rank 2

        Kapan Dipakai?

        • Kompetisi
        • Leaderboard
        • Ranking resmi (olahraga, lomba)

      • DENSE_RANK() — Ranking RAPAT

        Definisi

        Nilai sama → ranking sama
        Tanpa lompat angka

        Query

        DENSE_RANK() OVER (ORDER BY sales DESC)

        🔍 PROSES

        ProductSalesDense Rank
        D3001
        B2002
        C2002
        A1003

        📌 Tidak ada angka yang hilang

        Kapan Dipakai?

        • Kategori performa
        • Tier / level
        • Segmentasi

    • Tambahan

      PARTITION BY + RANKING (INI REAL ANALYST)

      Pertanyaan:

      “Ranking produk per kategori?”
      RANK() OVER (
    PARTITION BY category
    ORDER BY sales DESC
)

      📌 Ranking reset di setiap kategori

      Mental Model

      Category A → ranking sendiri
Category B → ranking sendiri

      Tidak pernah silang.

      Ranking ≠ Sorting

      🚨 Kesalahan besar pemula:

      “Aku sudah ORDER BY, berarti sudah ranking”

      ❌ SALAH

      • ORDER BY → urutan tampilan
      • RANK() → nilai peringkat (data)

      Ranking Function vs GROUP BY

      GROUP BYRanking
      MerangkumMembandingkan
      Hilang detailDetail utuh
      Satu baris per grupSatu baris per item

      Kesalahan Umum (Interview Trap)

      ❌ Lupa ORDER BY
      ❌ Salah pilih RANK vs DENSE_RANK
      ❌ Pakai ROW_NUMBER padahal nilai sama penting
      ❌ Bingung kenapa angka lompat

      Cara Memilih Ranking Function (RULE OF THUMB)

      Tanya diri kamu:
      1. Nilai sama boleh beda peringkat?
        • Ya → ROW_NUMBER
      1. Nilai sama → ranking sama?
        • Ya → lanjut
      1. Boleh lompat angka?
        • Ya → RANK
        • Tidak → DENSE_RANK

      Kesimpulan

      Ranking function memberi identitas posisi
      relatif antar baris
      TANPA merusak data asli.

  • LAG & LEAD (Time Comparison)

    Intuisi

    Bayangin kamu punya tabel berurutan (misalnya tanggal, ranking, waktu).

    • LAG → “lihat ke baris sebelumnya”
    • LEAD → “lihat ke baris berikutnya”

    👉 Tanpa menggabungkan tabel
    👉 Tanpa subquery
    👉 Tapi bisa “mengintip” data baris lain

    Definisi Formal

    • LAG

      Mengambil nilai dari baris sebelumnya dalam satu window.

    • LEAD

      Mengambil nilai dari baris sesudahnya dalam satu window.

    Kenapa Ini Penting buat Data Analyst?

    Karena analisis nyata sering butuh perbandingan antar waktu / urutan, misalnya:

    • Selisih penjualan hari ini vs kemarin
    • Apakah nilai naik atau turun
    • Deteksi lonjakan / penurunan
    • Growth, trend, perubahan perilaku

    👉 Tanpa LAG/LEAD, query jadi panjang + ribet

    • RUNNING TOTAL

      Intuisi

      Running Total = akumulasi berjalan

      “Total sampai titik ini”

      Contoh di kepala:

      • Hari 1 → 100
      • Hari 2 → 100 + 120 = 220
      • Hari 3 → 220 + 90 = 310

      📌 Bukan total keseluruhan, tapi total sampai baris tersebut

      Definisi

      Running Total adalah:

      Penjumlahan kumulatif nilai berdasarkan urutan tertentu

      Biasanya dipakai untuk:

      • Akumulasi penjualan
      • Akumulasi user
      • Saldo berjalan
      • Progress terhadap target

      Tabel Awal

      datesales
      2024-01-01100
      2024-01-02120
      2024-01-0390
      2024-01-04150

      Query Running Total

      SELECT
  date,
  sales,
  SUM(sales) OVER (
    ORDER BY date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS running_total
FROM sales;

      Proses yang Terjadi (visual)

      datesalesBaris yang dijumlahrunning_total

      01-0390100 + 120 + 90310

      📌 UNBOUNDED PRECEDING
      → dari baris paling awal

      📌 CURRENT ROW
      → sampai baris sekarang

      Versi Singkat (yang sering dipakai)

      SUM(sales) OVER (ORDER BY date)

      📌 Default frame:

      UNBOUNDED PRECEDING → CURRENT ROW

    • MOVING AVERAGE

      Intuisi

      Moving Average = rata-rata geser

      “Rata-rata beberapa periode terakhir”

      Dipakai buat:

      • Menghaluskan fluktuasi
      • Melihat tren
      • Analisis time series sederhana

      Definisi

      Moving Average adalah:

      Rata-rata nilai dalam window terbatas yang bergeser seiring baris

      Contoh: 3-Day Moving Average

      SELECT
  date,
  sales,
  AVG(sales) OVER (
    ORDER BY date
    ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
  ) AS moving_avg_3
FROM sales;

      Prosesnya (kelihatan jelas)

      📌 2 PRECEDING → 2 baris sebelum
      📌 CURRENT ROW → baris sekarang

      AVG(sales) OVER (
  ORDER BY date
  ROWS BETWEEN CURRENT ROW AND 2 FOLLOWING
)
      datesalesBaris dihitung
      01-01100100,120,90
      01-02120120,90,150

      AVG(sales) OVER (
  PARTITION BY store
  ORDER BY date
  ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
)

      RUNNING TOTAL vs MOVING AVERAGE

      SUM(sales) OVER ()

      🔹 Running total → “sampai sini totalnya berapa?”
      🔹 Moving average → “rata-rata terakhirnya berapa?”

Conditional Logic & Data Cleaning

Kenapa Conditional Logic & Cleaning Penting?

Di dunia nyata, data itu:

  • kotor
  • bolong (NULL)
  • gak konsisten
  • penuh label aneh

🎯 Tugas Data Analyst:

bikin data siap dianalisis, bukan cuma diambil.

Aspek

  • Conditional Logic
    • Definisi

      Conditional Logic di SQL adalah:

      Mekanisme untuk mengambil keputusan berdasarkan kondisi tertentu di dalam query.

      Kalau di kehidupan nyata:

      “Kalau hujan → pakai jas hujan

      Kalau tidak → pakai jaket”

      Di SQL, keputusan itu dibuat pakai CASE WHEN.

      Fungsi Utama Conditional Logic

      Conditional Logic dipakai untuk:

      1. Memberi label/kategori data
      1. Mengubah nilai berdasarkan kondisi
      1. Membuat aturan bisnis (business rules)
      1. Analisis bersyarat (conditional aggregation)
      1. Menandai data bermasalah

      📌 Intinya:

      Bukan membuang data, tapi membuat data “bermakna”

    • Alat Utama: CASE WHEN

      CASE WHEN adalah:

      Struktur if–else di SQL untuk menghasilkan nilai baru berdasarkan kondisi.

      Bentuk Umum

      CASE
  WHEN condition THEN result
  WHEN condition THEN result
  ELSE result
END

      📌 Catatan penting:

      • Dieksekusi atas ke bawah
      • Begitu kondisi pertama terpenuhi → berhenti

      Situasi Penggunaan CASE WHEN

      Digunakan ketika kamu ingin:

      KebutuhanGunakan
      Mengelompokkan dataCASE
      Memberi kategoriCASE
      Menentukan statusCASE
      Flag dataCASE
      Hitung bersyaratCASE
    • Contoh
      • Contoh 1 — Kategorisasi Data (Paling Umum)

        Situasi

        Seorang analyst ingin:

        Mengelompokkan pelanggan berdasarkan umur

        SQL

        SELECT
  name,
  age,
  CASE
    WHEN age < 18 THEN 'Underage'
    WHEN age BETWEEN 18 AND 30 THEN 'Young Adult'
    ELSE 'Adult'
  END AS age_group
FROM customers;

        🔍 Interpretasi

        • SQL membaca:
          • Kalau umur < 18 → Underage
          • Kalau 18–30 → Young Adult
          • Sisanya → Adult
        • Kolom baru age_group tercipta
        • Data tidak dihapus, hanya diberi makna

        📊 Hasil ini bisa langsung dipakai untuk:

        • Grafik
        • Segmentasi
        • Analisis perilaku

      • Contoh 2 — Conditional Logic vs WHERE

        Situasi

        Manager ingin melihat semua pelanggan, tapi tahu siapa yang aktif dan tidak

        ❌ SALAH (pakai WHERE)

        WHERE status = 'Active'

        👉 Data yang tidak aktif hilang

        ✅ BENAR (pakai CASE)

        SELECT
  name,

  CASE
    WHEN status = 'Active' THEN 'Aktif'

  END AS status_label
FROM customers;

        🔍 Interpretasi

        • Semua data tetap ada
        • Kita menambahkan informasi
        • Cocok untuk report & dashboard

  • NULL Handling
    • Apa itu NULL?

      NULL adalah:

      Kondisi di mana tidak ada nilai di dalam suatu kolom.

      NULL BUKAN:

      • 0
      • string kosong ''
      • false

      📌 NULL = unknown / missing / not recorded

      Kenapa NULL Berbahaya untuk Analyst?

      NULL bisa menyebabkan:

      • Hasil agregasi salah
      • Perhitungan gagal
      • Filter tidak bekerja sesuai dugaan
      • Insight keliru

      🧠 Prinsip penting:

      NULL itu menular → satu NULL bisa merusak satu perhitungan

      Cara Mengecek NULL

      ❌ Cara salah

      WHERE salary = NULL

      ✅ Cara benar

      WHERE salary IS NULL
WHERE salary IS NOT NULL

      📌 Ini aturan mutlak SQL.

    • Fungsi NULL Handling

      Dalam praktik data analyst, NULL handling dipakai untuk:

      1. Mengganti nilai kosong
      1. Menjaga agregasi tetap benar
      1. Menghindari error matematis
      1. Menandai data yang hilang
      1. Menjalankan aturan bisnis dengan aman

      Fungsi

      • COALESCE

        COALESCE adalah:

        Fungsi untuk mengambil nilai pertama yang tidak NULL

        Sintaks

        COALESCE(value1, value2, value3, ...)

        SQL akan membaca dari kiri ke kanan.

        Fungsi

        • Mengganti NULL
        • Menyediakan default value
        • Menjaga query tetap jalan

        Situasi Penggunaan

        • Kolom opsional (phone, discount)
        • Data dari berbagai sumber
        • Field yang sering kosong

        Contoh 1 — Teks

        SELECT
  name,
  COALESCE(phone, 'Tidak tersedia') AS phone
FROM customers;

        🔍 Interpretasi

        • Kalau phone NULL → diganti teks
        • Kalau tidak NULL → tampilkan nilai asli
        • Dashboard jadi rapi, gak bolong

        Contoh 2 — Numerik (WAJIB untuk agregasi)

        SELECT
  SUM(COALESCE(discount, 0)) AS total_discount
FROM orders;

        🔍 Interpretasi

        • Discount kosong dianggap 0
        • Total tidak jadi NULL
        • Hasil agregasi aman

        Contoh 3 — Multi fallback

        COALESCE(mobile_phone, home_phone, office_phone, 'Tidak ada nomor')

        📌 Dipakai saat data datang dari banyak sumber

      • NULLIF

        NULLIF adalah:

        Fungsi yang mengubah nilai tertentu menjadi NULL

        Sintaks

        NULLIF(value1, value2)

        Jika value1 = value2 → hasilnya NULL

        Fungsi

        • Mencegah error
        • Normalisasi data
        • Konversi nilai “palsu” jadi NULL

        Situasi Penggunaan

        • Menghindari pembagian dengan nol
        • Mengubah 0 / '' / -1 jadi NULL
        • Data hasil input manual

        Contoh 1 — Anti error pembagian

        SELECT
  revenue / NULLIF(quantity, 0)
FROM sales;

        🔍 Interpretasi

        • Kalau quantity = 0 → NULL
        • Tidak terjadi division by zero
        • Query tetap aman

        Contoh 2 — Normalisasi nilai

        SELECT
  NULLIF(score, 0) AS score
FROM exams;

        📌 Score 0 dianggap tidak ada nilai, bukan gagal

      • CASE + NULL

        Fungsi

        • Memberi label missing data
        • Kategorisasi data kosong
        • Validasi data

        Contoh 1 — Label data kosong

        SELECT
  name,
  CASE
    WHEN email IS NULL THEN 'Belum diisi'
    ELSE 'Sudah diisi'
  END AS email_status
FROM users;

        🔍 Interpretasi

        • Bukan mengisi data
        • Tapi menandai kondisi data
        • Cocok untuk monitoring kualitas data

        Contoh 2 — Gabungan CASE + COALESCE

        SELECT
  name,
  CASE
    WHEN COALESCE(age, 0) = 0 THEN 'Unknown'
    ELSE 'Known'
  END AS age_status
FROM users;

      • NULL dalam Agregasi

        Fakta penting:

        • COUNT(column) → mengabaikan NULL
        • COUNT(*) → menghitung semua baris

        Contoh

        SELECT
  COUNT(phone) AS with_phone,
  COUNT(*) AS total_data
FROM customers;

        🔍 Interpretasi

        • Selisih = jumlah NULL
        • Teknik cepat cek kualitas data

      • NULL & WHERE
        WHERE age > 30

        📌 Baris dengan age NULL tidak ikut

        👉 Kalau mau eksplisit:

        WHERE age > 30 OR age IS NULL

      • Best Practice NULL Handling (Industri)
        1. Jangan langsung hapus data NULL
        1. Tandai dulu (CASE)
        1. Ganti NULL hanya jika secara bisnis masuk akal
        1. Gunakan COALESCE sebelum agregasi
        1. Gunakan NULLIF untuk anti error

  • Data Cleaning Text
    • Konsep

      Data Cleaning Teks adalah:

      Proses menormalkan dan merapikan data bertipe string agar konsisten dan bisa dianalisis dengan benar.

      📌 Masalah teks biasanya tidak kelihatan, tapi efeknya fatal.

      Masalah Teks yang Sering Terjadi

      MasalahContoh
      Spasi tersembunyi' Jakarta', 'Bandung '
      Kapital tidak konsistenjakarta, Jakarta, JAKARTA
      Format tidak seragam0812-xxx, 0812 xxx
      Typo ringanMale, male, MALE
    • Fungsi
      • TRIM / LTRIM / RTRIM

        TRIM adalah:

        Fungsi untuk menghapus spasi di awal dan akhir teks

        Berfungsi untuk:

        • Membersihkan input manual
        • Menyelamatkan JOIN & GROUP BY
        • Normalisasi data kategori

        Digunakan ketika:

        • Data hasil input user
        • Import CSV / Excel
        • Data survei

        Contoh

        SELECT
  TRIM(city) AS city_clean
FROM customers;

        🔍 Interpretasi

        • ' Jakarta ' → 'Jakarta'
        • Spasi tersembunyi hilang
        • Data siap dipakai

        LTRIM / RTRIM

        LTRIM(city)  -- hapus spasi kiri
RTRIM(city)  -- hapus spasi kanan

        📌 Jarang dipakai, tapi penting diketahui

      • UPPER / LOWER / INITCAP
        • UPPER → semua huruf besar
        • LOWER → semua huruf kecil

        Fungsi

        • Konsistensi teks
        • Anti duplikasi kategori
        • Aman untuk JOIN

        Situasi Penggunaan

        • Nama kota
        • Gender
        • Status
        • Email

        Contoh

        SELECT
  UPPER(city) AS city
FROM customers;

        🔍 Interpretasi

        • jakarta, Jakarta, JAKARTA → JAKARTA
        • GROUP BY jadi bersih

        Contoh JOIN Aman

        ON UPPER(a.city) = UPPER(b.city)

        📌 Ini best practice industri

      • REPLACE

        REPLACE adalah:

        Fungsi untuk mengganti bagian teks tertentu

        Sintaks

        REPLACE(teks, yang_dicari, pengganti)

        Fungsi

        • Membersihkan format
        • Hilangkan simbol
        • Standarisasi nilai

        Situasi Penggunaan

        • Nomor HP
        • Kode produk
        • Identifier

        Contoh

        SELECT
  REPLACE(phone, '-', '') AS phone_clean
FROM customers;

        Interpretasi

        • 0812-345-678 → 0812345678
        • Format jadi konsisten

        Contoh Lanjutan

        REPLACE(REPLACE(phone, '-', ''), ' ', '')

        📌 Bisa di-chain (berantai)

      • SUBSTRING / LEFT / RIGHT

        Mengambil bagian tertentu dari teks

        Contoh

        SUBSTRING(code, 1, 3)

        📌 Ambil 3 karakter pertama

        Situasi Penggunaan

        • Ambil kode wilayah
        • Ekstrak prefix
        • Parsing ID

        Contoh

        SELECT
  SUBSTRING(product_code, 1, 3) AS category_code
FROM products;

      • LENGTH

        Menghitung jumlah karakter

        Fungsi

        • Validasi data
        • Deteksi error input

        Contoh

        SELECT
  phone
FROM customers
WHERE LENGTH(phone) < 10;

        🔍 Interpretasi

        • Nomor HP terlalu pendek
        • Data mencurigakan

      • CONCAT / ||

        Menggabungkan teks

        Contoh

        SELECT
  CONCAT(first_name, ' ', last_name) AS full_name
FROM users;

        📌 Di beberapa DB:

        first_name || ' ' || last_name

    • Pola Umum

      Pipeline Cleaning (Best Practice)

      UPPER(TRIM(city))

      Urutan logis:

      • TRIM dulu
      • Baru normalisasi huruf

      Contoh Real Case (Lengkap)

      • Situasi

        Data kota kacau:

        • Spasi
        • Kapital beda
        • NULL
      • SQL
        SELECT
  COALESCE(UPPER(TRIM(city)), 'UNKNOWN') AS city_clean
FROM customers;

        Interpretasi

        • Spasi dibuang
        • Huruf diseragamkan
        • NULL ditangani
        • Siap dipakai untuk GROUP BY

      Kesalahan Umum Analyst Pemula

      ❌ Langsung GROUP BY city

      ❌ JOIN tanpa normalisasi

      ❌ Asumsi teks selalu rapi

      ✅ Cleaning dulu → baru analisis

  • Data Validation & Flagging
    • Konsep

      Data Validation & Flagging adalah:

      Proses mengecek apakah data masuk akal, lalu memberi tanda (flag) pada data yang bermasalah atau perlu perhatian.

      📌 Penting:

      • Tidak langsung menghapus data
      • Data tetap ada, tapi diberi konteks

      Kenapa Analyst Perlu Validation & Flagging?

      Tanpa validasi:

      • Insight bisa salah
      • KPI menipu
      • Keputusan bisnis keliru

      Dengan flagging:

      • Data transparan
      • Diskusi dengan stakeholder lebih jelas
      • Bisa audit data

      🧠 Mindset:

      “Data boleh jelek, asal jujur.”

      Kapan Data Perlu Divalidasi?

      Umumnya perlu divalidasi kalau:

      • Data hasil input manual
      • Data dari banyak sumber
      • Data historis lama
      • Data keuangan / demografi

      Alat Utama - CASE WHEN untuk Validasi

      Berfungsi untuk:

      • Memberi status data
      • Menandai outlier
      • Menentukan kelayakan data

      Bentuk umum

      CASE
  WHEN kondisi_salah THEN 'Invalid'
  WHEN kondisi_mencurigakan THEN 'Suspicious'
  ELSE 'Valid'
END

    • Contoh
      • Contoh 1 — Validasi Nilai Numerik

        🎯 Situasi

        Kolom age hasil input user:

        • Bisa negatif
        • Bisa terlalu besar

        SQL

        SELECT
  name,
  age,
  CASE
    WHEN age < 0 THEN 'Invalid'
    WHEN age > 100 THEN 'Suspicious'
    ELSE 'Valid'
  END AS age_flag
FROM customers;

        🔍 Interpretasi

        • Invalid → jelas salah
        • Suspicious → perlu dicek
        • Valid → aman dianalisis

        📌 Data tidak dihapus, cuma diberi label.

      • Contoh 2 — Validasi Data Kosong (Completeness)

        🎯 Situasi

        Email wajib diisi, tapi banyak NULL.

        SQL

        SELECT
  user_id,
  CASE
    WHEN email IS NULL THEN 'Missing'
    ELSE 'Complete'
  END AS email_status
FROM users;

        🔍 Interpretasi

        • Bisa dihitung berapa % data lengkap
        • Cocok untuk data quality report

      • Contoh 3 — Validasi Rentang Nilai (Range Check)

        🎯 Situasi

        Nilai ujian 0–100.

        SQL

        SELECT
  student_id,
  score,
  CASE
    WHEN score < 0 OR score > 100 THEN 'Out of Range'
    ELSE 'Valid'
  END AS score_flag
FROM exam_results;

        🔍 Interpretasi

        • Data di luar aturan bisnis terdeteksi
        • Aman untuk analisis lanjutan

    • VALIDASI KATEGORI & TEKS
      • Validasi Kategori Tidak Dikenal

        🎯 Situasi

        Kolom gender seharusnya cuma M atau F.

        SQL

        SELECT
  gender,
  CASE
    WHEN gender IN ('M', 'F') THEN 'Valid'
    ELSE 'Invalid'
  END AS gender_flag
FROM users;

        🔍 Interpretasi

        • Typo langsung ketahuan
        • Bisa lapor ke tim data source

      • Validasi Panjang Teks (Format)

        🎯 Situasi

        Nomor HP minimal 10 digit.

        SQL

        SELECT
  phone,
  CASE
    WHEN phone IS NULL THEN 'Missing'
    WHEN LENGTH(phone) < 10 THEN 'Too Short'
    ELSE 'Valid'
  END AS phone_flag
FROM customers;

        🔍 Interpretasi

        • Lebih informatif daripada sekadar NULL check
        • Cocok untuk CRM / data kontak

    • BUSINESS RULE VALIDATION

      🎯 Situasi

      Transaksi sukses harus punya tanggal pembayaran.

      SQL

      SELECT
  order_id,
  CASE
    WHEN status = 'Success' AND paid_date IS NULL THEN 'Violation'
    ELSE 'OK'
  END AS payment_rule_flag
FROM orders;

      🔍 Interpretasi

      • Data melanggar aturan bisnis terdeteksi
      • Penting untuk audit & finance

    • FLAGGING UNTUK ANALISIS (BUKAN HAPUS)

      Flagging untuk Filtering Belakangan

      • Situasi

        Jangan buang data sekarang, filter nanti.

      • SQL
        SELECT
  *,
  CASE
    WHEN age < 0 OR age > 100 THEN 0
    ELSE 1
  END AS is_valid
FROM customers;

        🔍 Interpretasi

        • is_valid = 1 → boleh dipakai
        • is_valid = 0 → exclude di tahap akhir

        📌 Fleksibel & aman.

    • DATA QUALITY METRICS

      Hitung Kualitas Data

      • Situasi

        Berapa % data valid?

      • SQL
        SELECT
  COUNT(*) AS total_data,
  COUNT(CASE WHEN age BETWEEN 0 AND 100 THEN 1 END) AS valid_age
FROM customers;

        🔍 Interpretasi

        • Bisa bikin KPI data quality
        • Dipakai di data governance

    • Best Practice Industri
      1. Flag dulu, jangan hapus
      1. Pisahkan:
        • Invalid
        • Suspicious
        • Valid
      1. Dokumentasikan aturan validasi
      1. Validasi sebelum agregasi

  • Real Analysis Pattern

    Definisi

    Pola-pola query SQL yang sering muncul di pekerjaan data analyst, hasil gabungan:
    • conditional logic
    • NULL handling
    • cleaning
    • agregasi

    📌 Ini yang bikin:

    • dashboard masuk akal
    • insight dipercaya
    • interview lolos 😄

    PATTERN

    • Conditional Aggregation

      Situasi

      Hitung metrik berbeda dari tabel yang sama dalam satu query

      Pola umum

      AGG_FUNC(
  CASE
    WHEN condition THEN value
    ELSE 0 / NULL
  END
)

      Contoh

      SELECT
  COUNT(*) AS total_tx,
  COUNT(CASE WHEN status = 'Success' THEN 1 END) AS success_tx,
  COUNT(CASE WHEN status = 'Failed' THEN 1 END) AS failed_tx
FROM transactions;

      🔍 Interpretasi

      • Satu tabel → banyak KPI
      • Efisien
      • Sangat umum di dashboard

    • TOP-N PER GROUP

      Situasi

      Cari produk terlaris di setiap kategori

      Pola (Window Function)

      ROW_NUMBER() OVER (
  PARTITION BY group_col
  ORDER BY metric DESC
)

      Contoh

      SELECT *
FROM (
  SELECT
    category,
    product,
    SUM(sales) AS total_sales,
    ROW_NUMBER() OVER (
      PARTITION BY category
      ORDER BY SUM(sales) DESC
    ) AS rn
  FROM orders
  GROUP BY category, product
) t
WHERE rn = 1;

      🔍 Interpretasi

      • Ranking dilakukan per kategori
      • Hasil = top product masing-masing kategori
      • Ini query favorit interviewer

    • DEDUPLICATION (HAPUS DUPLIKAT)

      Situasi

      Data user dobel, ambil yang paling baru

      Pola

      ROW_NUMBER() OVER (
  PARTITION BY unique_key
  ORDER BY updated_at DESC
)

      Contoh

      SELECT *
FROM (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY email
      ORDER BY updated_at DESC
    ) AS rn
  FROM users
) t
WHERE rn = 1;

      🔍 Interpretasi

      • Satu email → satu baris
      • Data terbaru dipilih
      • Aman untuk analisis

    • ANTI-JOIN (DATA YANG TIDAK ADA)

      Situasi

      Cari user yang belum pernah transaksi

      Pola

      LEFT JOIN + IS NULL

      Contoh

      SELECT u.user_id
FROM users u
LEFT JOIN transactions t
  ON u.user_id = t.user_id
WHERE t.user_id IS NULL;

      🔍 Interpretasi

      • LEFT JOIN ambil semua user
      • NULL artinya tidak ada pasangan
      • Dipakai untuk:
        • churn
        • inactive users

    • FUNNEL ANALYSIS (DASAR)

      Situasi

      Dari view → add to cart → purchase

      Contoh

      SELECT
  COUNT(DISTINCT user_id) AS total_users,
  COUNT(DISTINCT CASE WHEN step = 'view' THEN user_id END) AS view_users,
  COUNT(DISTINCT CASE WHEN step = 'cart' THEN user_id END) AS cart_users,
  COUNT(DISTINCT CASE WHEN step = 'purchase' THEN user_id END) AS purchase_users
FROM user_events;

      🔍 Interpretasi

      • Drop-off tiap tahap terlihat
      • KPI produk & marketing

    • COHORT ANALYSIS (BASIC)

      Situasi

      User dikelompokkan berdasarkan bulan join

      Contoh

      SELECT
  DATE_TRUNC('month', signup_date) AS cohort,
  COUNT(*) AS users
FROM users
GROUP BY cohort
ORDER BY cohort;

      🔍 Interpretasi

      • Cohort = waktu pertama kali join
      • Dipakai untuk retention & growth

    • DATA QUALITY SUMMARY

      Situasi

      Report kondisi data

      Contoh

      SELECT
  COUNT(*) AS total,
  COUNT(email) AS with_email,
  COUNT(CASE WHEN age BETWEEN 0 AND 100 THEN 1 END) AS valid_age
FROM users;

      🔍 Interpretasi

      • Stakeholder tahu kualitas data
      • Analyst transparan

    • SAFE METRICS (ANTI ERROR)

      Situasi

      Hitung rata-rata aman dari error

      Contoh

      SELECT
  SUM(revenue) / NULLIF(SUM(quantity), 0) AS avg_price
FROM sales;

      🔍 Interpretasi

      • Tidak error
      • Lebih robust
      • Best practice industri

Date & Time Analysis
  • Konsep

    Analisis data berbasis waktu, untuk menjawab pertanyaan seperti:

    • Kapan sesuatu terjadi?
    • Bagaimana perubahan dari waktu ke waktu?
    • Ada pola harian / bulanan / musiman?
    • Dibandingkan periode sebelumnya, naik atau turun?

    📌 Hampir SEMUA analisis bisnis punya dimensi waktu.

    Tipe Data Waktu (Umum)

    TipeContohKegunaan
    DATE2025-01-15Analisis harian
    TIME14:30:00Jam transaksi
    DATETIME / TIMESTAMP2025-01-15 14:30:00Event log
    TIMESTAMP WITH TZUTC+7Data global
  • Extracting Date Parts

    Fungsi Umum

    (Nama bisa beda tergantung DB, konsepnya sama)

    TujuanFungsi
    TahunYEAR(date)
    BulanMONTH(date)
    HariDAY(date)
    JamHOUR(datetime)
    Hari dalam mingguDAYOFWEEK()

    Situasi Penggunaan

    • Sales per bulan
    • Aktivitas per jam
    • Pola hari kerja vs weekend

    Contoh

    SELECT
    order_date,
    YEAR(order_date)  AS year,
    MONTH(order_date) AS month,
    DAY(order_date)   AS day
FROM orders;

    Interpretasi

    • Memecah satu kolom tanggal
    • Siap untuk agregasi & grouping

  • GROUPING by Time (Trend Analysis)

    Masalah Umum

    “Kenapa GROUP BY order_date terlalu detail?”

    Karena:

    • Data bisa ribuan baris per hari
    • Analyst biasanya butuh bulan / minggu

    Contoh: Sales Bulanan

    SELECT
    YEAR(order_date)  AS year,
    MONTH(order_date) AS month,
    SUM(amount) AS total_sales
FROM orders
GROUP BY year, month
ORDER BY year, month;

    Interpretasi

    • Melihat tren penjualan dari bulan ke bulan

  • Date Filtering (Time-based WHERE)

    Situasi Penggunaan

    • Data bulan ini
    • 30 hari terakhir
    • Tahun tertentu

    Contoh: Data Tahun 2025

    SELECT *
FROM orders
WHERE YEAR(order_date) = 2025;

    📌 Catatan Analyst:

    Ini mudah dibaca, tapi kadang kurang optimal untuk index.

    Versi Lebih Aman

    WHERE order_date >= '2025-01-01'
  AND order_date <  '2026-01-01'

  • Date Arithmetic (Selisih Waktu)

    Fungsi Umum

    TujuanFungsi
    Selisih hariDATEDIFF()
    Tambah tanggalDATE_ADD()
    Kurang tanggalDATE_SUB()

    Contoh: Lama Proses Order

    SELECT
    order_id,
    DATEDIFF(delivery_date, order_date) AS delivery_days
FROM orders;

    Interpretasi

    • Mengukur SLA
    • Deteksi keterlambatan

  • Current Date & Relative Time

    Fungsi Umum

    FungsiMakna
    CURRENT_DATETanggal hari ini
    NOW()Tanggal + waktu
    CURRENT_TIMESTAMPTimestamp

    Contoh: 7 Hari Terakhir

    SELECT *
FROM orders
WHERE order_date >= CURRENT_DATE - INTERVAL 7 DAY;

    Interpretasi

    • Analisis data terbaru
    • Cocok untuk dashboard harian

  • Time-Based Bucketing

    Masalah

    “Dataku terlalu detail, mau diringkas”

    Contoh: Jam Transaksi

    SELECT
    HOUR(order_time) AS hour,
    COUNT(*) AS total_orders
FROM orders
GROUP BY hour
ORDER BY hour;

    Interpretasi

    • Mengetahui jam ramai transaksi
    • Berguna untuk operasional

  • Weekend vs Weekday Analysis
    SELECT
    CASE
        WHEN DAYOFWEEK(order_date) IN (1,7) THEN 'Weekend'
        ELSE 'Weekday'
    END AS day_type,
    COUNT(*) AS total_orders
FROM orders
GROUP BY day_type;

    Interpretasi

    • Perilaku user beda di hari kerja vs libur

  • Growth Analysis (Tanpa Window vs Dengan Window)

    ❌ TANPA Window (ribet)

    SELECT
    curr.month,
    curr.total_sales - prev.total_sales AS growth
FROM monthly_sales curr
JOIN monthly_sales prev
  ON curr.month = prev.month + 1;

    ✅ DENGAN Window (ideal)

    SELECT
    month,
    total_sales,
    total_sales - LAG(total_sales) OVER (ORDER BY month) AS growth
FROM monthly_sales;
